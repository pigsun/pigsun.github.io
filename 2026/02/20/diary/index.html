从零搭建个人博客：Hexo + GitHub Pages 的实践与思考

一、为什么选择 Hexo？

我最终选择了 Hexo + GitHub Pages 作为博客方案，核心原因有三：

  1. 成本为 0（GitHub Pages 免费托管）
  2. 纯静态部署，无服务器维护压力
  3. Markdown 驱动，专注内容本身

从技术本质来看，这是一个典型的 SSG（Static Site Generator）架构流程：

Hexo 本质是一个基于 Node.js 的静态网站生成器。
Markdown 文件经过渲染引擎转换为 HTML，再通过主题模板进行结构化封装，最终生成完整网页。

这次搭建让我第一次真正理解了“静态网站生成”的底层逻辑。

二、搭建过程中遇到的关键问题

1️⃣ Node 与 Hexo 版本兼容问题

问题表现

安装 Hexo CLI 后执行命令报错，依赖冲突无法正常运行。

根本原因

Hexo 某些依赖对 Node 版本存在要求，而当前 Node 版本不兼容。

解决方案

使用 nvm 管理多个 Node 版本：

通过版本隔离，解决依赖冲突问题。

🔎 技术收获：

  • 理解 Node 生态对版本敏感
  • 学会使用版本管理工具
  • 意识到环境管理是工程开发的基础能力

2️⃣ Markdown 图像渲染问题

初期 Markdown 中插入图片无法正常加载。

解决方式

安装渲染插件：

  • hexo-renderer-marked
  • 相关 markdown 依赖

这让我意识到：

Hexo 并不是“自动支持所有功能”，很多能力来自插件生态。

📌 本质理解：

Hexo 是核心框架 + 插件驱动架构。

3️⃣ 主题更换与模板渲染机制

我选择了 Hexo Theme Butterfly 主题。

该主题采用 Pug 模板引擎进行渲染。

流程变成：

这让我第一次理解：

  • 模板引擎的作用
  • 视图层如何抽象
  • 前端结构与内容如何分离

🔎 技术认知升级：

  • 博客不只是“写文章”
  • 本质是“内容数据 + 模板引擎 + 构建工具”

三、这次实践真正带给我的成长

表面上是搭建博客，实际上我理解了：

层级 收获
环境层 Node 版本管理
构建层 静态网站生成流程
渲染层 模板引擎机制
部署层 GitHub Pages 自动托管

这次经历让我对“前端工程化”有了初步认知。

过去我认为：

写 Markdown = 写博客

现在我理解：

博客是一个工程系统

个人博客的搭建

今天搭建了自己的博客，用的是Hexo+Github Page,虽然只是静态markdown文件的简单的渲染，但是再借助ai帮助的情况下还是花了不少时间。

懂得了原理，描述一下就是原生md文件在node环境下，通过Hexo框架渲染成web。

遇到的难点：node和hexo之间版本的兼容问题，在这过程中我下载了nvm，来管理多个node版本。

图像嵌入的加载，下载了插件hexo-renderer-marked以及相关依赖，让我可以非常简便插入图像

主题的更换，同样下载主题（butterfly）与渲染依赖（因为主题butterfly是渲染成pug文件，通过相关依赖才得以生成web）
