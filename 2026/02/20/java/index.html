数据类型转换

隐形转换

隐式类型转换支持字节数小的类型自动转换为字节数大的类型，整数类型自动转换为小数类型，转换规则如下：【小范围转大范围】

  • byte→short(char)→int→long→float→double
  • 问题：为什么long比float大，还能转换为float呢？
  • 答：位数大小（bit size） 和 数值范围（range of values） 是两个不同的概念。
    • long: 是一个 64位 (bit) 的有符号整数类型，范围是连续的，在这个范围内的每一个整数都可以被精确表示
    • float: 是一个 32位 (bit) 的单精度浮点数类型，IEEE 754 标准，范围是极其巨大的，但是不连续的
    • 总结：Java 在设计类型转换规则时，判断“扩大”的标准是看目标类型的值域是否完全包含了源类型的值域。由于 float 的最大值远大于 long 的最大值，所以从 long 到 float 被认为是一种扩大转换。虽然这种转换会带来精度上的损失，但不会导致数值溢出到无效范围

显性转换

也叫强转换类型，牺牲精度强行进行类型转换 【大范围转小范围】

数据类型自动提升

这个特性是由 Java虚拟机规范 定义的，也是为了提高运行的效率。其他的特性还有：

  • 如果一个操作数是long型，计算结果就是long型
  • 如果一个操作数是float型，计算结果就是float型
  • 如果一个操作数是double型，计算结果就是double型
  • 优先级（double > float > long > int）

面向对象编程的三大特性。

封装

封装思想其实就是把实现细节给隐藏了，外部只需知道这个方法是什么作用，而无需关心如何实现。外界只能调用接口or使用该方法，这样将操作成员变量的权限与外界隔开。

目的：是为了保证变量的安全性，使用者不必在意具体实现细节，而只是通过外部接口即可访问类的成员，如果不进行封装，类中的实例变量可以直接查看和修改，可能给整个代码带来不好的影响，因此在编写类时一般将成员变量私有化，外部类需要同getter和setter方法来查看和设置变量。【小结：成员变量应该私有化（private），使外部只能通过getter、setter方法来查看和设置变量】

即变量私有化，操作公开化

继承

继承允许一个类（子类/派生类）获得另一个类（父类/基类/超类）的属性和方法。这是面向对象编程中实现代码复用和建立类之间层次关系的关键机制。

关键要点

  1. 关键字 extends：子类通过 class 子类名 extends 父类名 的语法来继承父类。
  2. 成员访问
    • 子类可以直接访问父类中 public 和 protected 修饰的成员（属性和方法）。
    • 子类拥有父类的 private 私有成员，但不能直接访问。需要通过父类提供的 public 或 protected 的 getter/setter 方法来间接访问。
  3. 单一继承：Java 不支持多继承（一个类不能同时继承多个类），但支持多层继承（A 继承 B，B 继承 C）。
  4. super 关键字
    • super 指代当前对象的父类。
    • super.成员变量：访问父类的成员变量。
    • super.方法名()：调用父类的方法。
    • super(...)：必须出现在子类构造方法的第一行，用于调用父类的构造方法。如果父类有有参构造方法，子类必须显式调用 super(参数)。如果父类只有无参构造方法，super() 可以省略，编译器会默认添加

多态

多态是指同一个方法调用，由于对象的实际类型不同，会产生不同的行为表现。它极大地提高了程序的扩展性和可维护性

实现多态的三个必要条件

  1. 继承 (Inheritance)：必须存在父子类关系。
  2. 重写 (Overriding)：子类必须重写父类的方法，以提供自己特定的实现。
  3. 向上转型 (Upcasting)：父类的引用变量可以指向其子类的实例。

关键要点

  1. 方法重写 (Override) vs 方法重载 (Overload)：

    • 重写 (Override)：发生在父子类之间，方法名、参数列表、返回值类型完全相同（或返回值是子类型）。使用 @Override 注解可以帮助检查。
    • 重载 (Overload)：发生在同一个类中，方法名相同，但参数列表不同。
    • 静态方法：不能被重写。父类和子类中同名的静态方法是各自独立的。

​ 3.向下转型 (Downcasting)：将父类引用强制转换回子类引用，以便调用子类特有的方法。这需要使用 instanceof 进行安全检查，以防 ClassCastException。

  1. instanceof 关键字：用于在运行时判断一个对象是否是某个类（或其子类）的实例。

总结

  • 继承是基础，它定义了类之间的“是什么”(IS-A) 关系，实现了代码复用。
  • 多态是目的，它利用继承和重写，实现了接口的通用性，让代码更加灵活和易于扩展。

final

1. 修饰变量 (final variable)

  • 基本数据类型：变量的值不能被改变。
  • 引用数据类型：变量的引用地址不能被改变，但所指向对象的内部状态是可以修改的。

2. 修饰方法 (final method)

  • 该方法不能被子类重写（Override）。
  • 目的：保护方法的实现不被修改，通常用于核心、关键的算法或逻辑。

3. 修饰类 (final class)

  • 该类不能被继承。
  • 目的：防止类被扩展，保证类的完整性。Java 中的 String, Integer 等包装类都是 final 类。

抽象类（Abstract Class）

关键特征

  1. abstract 修饰符：定义抽象类和抽象方法都必须使用 abstract 关键字。
  2. 不能被实例化：不能直接使用 new 来创建抽象类的对象。

​ 3.可以包含普通成员：抽象类不仅可以包含抽象方法，还可以包含普通的成员变量、普通方法、构造方法等。

​ 4.强制子类实现：任何继承抽象类的非抽象子类，都必须实现其所有的抽象方法，除非子类本身也是抽象类。

​ 5.匿名内部类实现：可以创建抽象类的匿名子类来实现其抽象方法。

接口 (Interface)

接口是比抽象类更纯粹的抽象，它代表着一种“能力”或“契约”。

关键特征

  1. interface 修饰符：定义接口使用 interface 关键字。

  2. 纯抽象：在 Java 8 以前，接口中的所有方法都是 public abstract 的（这两个关键字可以省略）。

  3. 常量：接口中定义的变量默认是 public static final 的，即公共、静态、不可变的常量。

  4. default 方法 (Java 8+)：可以为接口方法提供一个默认实现，这样实现类就不强制必须重写这个方法。

  5. 实现接口：类使用 implements 关键字来实现接口。一个类可以实现多个接口，从而具备多种能力。

  6. 多态性：接口引用可以指向任何实现了该接口的类的实例。（这个特性让接口成为 Java 中实现松耦合、高扩展性架构的基石）

    为什么这很重要？

    1. 解耦合 (Decoupling)：drawShape 方法只知道它接收的是一个 Drawable，它并不关心这个对象具体是 Circle 还是 Rectangle。这就将方法的实现与具体的实现类解耦了。
    2. 可扩展性 (Extensibility)：如果未来需要添加一个新的图形，比如 Star，我们只需要让 Star 类实现 Drawable 接口，然后就可以直接将 Star 的实例传给 drawShape 方法，而无需修改 drawShape 的代码。

    接口引用指向实现类实例的特性，是接口实现“多态”的核心机制，它让我们的代码更加灵活、健壮和易于维护。

匿名内部类

匿名内部类的核心思想：当一个类的实现（或方法的重写）只使用一次，并且实现逻辑相对简单时，可以使用匿名内部类来快速、简洁地创建一个对象，而无需为其定义一个正式的、有名字的类。

优点：代码简洁，逻辑内聚，适用于一次性任务。
缺点：无法复用，不适合复杂逻辑，过多使用可能影响代码可读性。

包装类

其核心意义——桥接基本数据类型和面向对象世界

为什么需要包装类？

Java 的基本数据类型（int, double, boolean 等）不是对象，它们是直接存储在栈中的原始值。这与 Java “一切皆对象”的面向对象思想有所冲突。

  1. 面向对象的需求：有时候我们需要将一个整数作为一个对象来处理。例如，将它作为方法的参数传递给期望接收对象的方法，或者把它存储在只接受对象的集合（如 List）中。
  2. 提供工具方法：包装类提供了许多实用的静态方法，用于在字符串和基本类型之间进行转换，或者获取类型的相关信息。

Java异常机制

异常机制分为异常（Exception）和错误（Error）

异常（Exception）

运行时异常

定义：在编译阶段无法感知代码是否会出现问题，只有在运行的时候才知道会不会出错（正常情况下是不会出错的），这样的异常称为运行时异常。

特点：所有的运行时异常都直接继承自RuntimeException（RuntimeException也是继承Exception）

编译时异常

定义：在编译阶段就需要进行处理的异常（捕获异常）如果不进行处理，将无法通过编译！

特点：默认直接继承自Exception类的异常都是编译时异常

错误（Error）

错误比异常更严重，异常就是不同寻常，但不一定会导致致命的问题，而错误是致命问题，一般出现错误可能JVM就无法继续正常运行了

异常处理

程序出现异常时，默认会交给JVM来处理，JVM发现任何异常都会立即终止程序运行，并在控制台打印栈追踪信息。这是就需要编写程序手动捕获异常，使程序继续正常运行。(一旦手动捕获后,异常就不再交给JVM处理)

运行时异常在编译时可以不用捕获，但是编译时异常必须进行处理

泛型

核心目的：泛型（Generics）的核心目的是在编译期就发现类型不匹配的问题，从而避免在运行时发生 ClassCastException。

类型擦除 (Type Erasure)

这是泛型实现的关键机制。正如您所说，泛型信息在编译后会被擦除。

  • 编译前：Score<String>, Score<Integer>
  • 编译后：Score，所有 T 都被替换为它的上界（通常是 Object）。

关键点：虽然类型被擦除了，但编译器会在必要的地方自动插入类型转换代码，确保类型安全。

泛型的使用细节

  • 类上使用：public class MyClass<T> {}

  • 方法上使用：

    • 依赖于类：public T getValue() {} (这个 T 从属于类的 <T>)
    • 独立于类 (自定义)：public <E> void printElement(E element) {} (这个 <E> 是方法独有的)

  • 静态成员：静态成员属于类本身，而不是实例。在类加载时，泛型信息还未确定，因此静态变量不能使用类的泛型参数。但静态方法可以有自己的泛型参数。

  • 基本类型：泛型不能是 int, double, boolean 等基本数据类型，必须是它们的包装类 Integer, Double, Boolean。

I/O流

Java I/O 的设计大量使用了装饰者模式 (Decorator Pattern)。其核心思想是：

  1. 基础流 (Base Stream)：如 FileInputStream / FileOutputStream / FileReader / FileWriter，负责与数据源（如文件）建立连接。

  2. 装饰流 (Decorating Stream)：如 BufferedInputStream / InputStreamReader / ObjectInputStream 等，它们接收一个基础流作为构造参数，为其“装饰”上新的功能（如缓冲、字符转换、对象序列化等）。

    小结：

    除文件流外的五个流只能嵌套文件流，他们之间无法相互嵌套，所以用流的时候需要判断应该什么流嵌套文件流

这种模式使得 I/O 流的功能可以像搭积木一样灵活组合，非常强大。

基础：文件流 (File Streams)

直接操作文件的流

  • 文件字节流：FileInputStream / FileOutputStream
    • 适用场景：所有类型的文件（图片、视频、文本等）。数据以 byte 为单位读写。
    • 注意：处理文本文件时，需要考虑字符编码问题（一个中文字符可能占多个字节）。
  • 文件字符流：FileReader / FileWriter
    • 适用场景：纯文本文件。数据以 char 为单位读写，内部自动处理了字符编码转换。
    • 注意：没有 available() 方法。

转换流 (Conversion Streams) - 解决编码问题

这是 I/O 流中非常重要的一个环节，它连接了字节流和字符流。

  • InputStreamReader：将字节输入流（如 FileInputStream）转换为字符输入流（Reader）。
  • OutputStreamWriter：将字符输出流（Writer）转换为字节输出流（如 FileOutputStream）。

意义：允许我们在处理文件时，可以指定字符集（如 UTF-8, GBK），解决了乱码问题。

缓冲流 (Buffered Streams) - 提升性能

对基础流进行性能优化

  • 原理：在内存中开辟一个缓冲区。当从流中读取数据时，不是每次只读一个字节/字符，而是先将数据批量读入缓冲区，程序再从缓冲区读取。写入同理，先写入缓冲区，再批量写入目标。这大大减少了实际的 I/O 操作次数。
  • 类型：BufferedInputStream / BufferedOutputStream / BufferedReader / BufferedWriter
  • mark() 和 reset()：BufferedInputStream 和 BufferedReader 提供了这两个方法，允许你在读取流的过程中标记一个位置，之后可以调用 reset() 返回到该位置。这在解析数据时非常有用。

模块 A：短链接跳转 & 缓存设计

A1. ⭐ 短链接是怎么生成的？

答：
对原始 URL 拼接一个随机 UUID 后，使用 MurmurHash 算法计算哈希值，再转换为 Base62 编码（0-9a-zA-Z），截取前 6 位作为短链接后缀。之所以拼接 UUID 是为了每次生成不同的哈希输入，降低冲突概率。生成后检查布隆过滤器是否已存在，存在则重试，最多重试 10 次。

A2. ⭐⭐ 为什么选布隆过滤器而不是直接查数据库判断短链接是否重复？

答：
布隆过滤器是纯内存操作，查询时间复杂度 O(k)（k 为哈希函数数量），性能远超数据库查询。短链接系统中，短链接的数量可能达到亿级，每次创建都查一次数据库压力极大。布隆过滤器的缺点是有一定误判率（可能误判为存在），但不会漏判（不存在一定返回不存在），对于防重场景完全够用，误判时最多多重试一次。

A3. ⭐⭐ 短链接跳转时是如何防止缓存穿透的？

答：
使用了两层防护：

  1. 布隆过滤器：存储所有有效的完整短链接（fullShortUrl），跳转时先查布隆过滤器，如果不存在直接返回 404，不会访问数据库。
  2. 空值缓存：对于布隆过滤器误判通过但数据库中不存在（或已过期）的短链接，在 Redis 中写入一个空值缓存（key = GOTO_IS_NULL:{fullShortUrl}, value = “-“），TTL 30 分钟，后续请求直接返回 404，不再查库。

A4. ⭐⭐ 短链接跳转时是如何防止缓存击穿的？

答：
使用 Redisson 分布式锁。当缓存中没有该短链接时，多个并发请求同时到达，只有第一个请求能获取到锁并查询数据库重建缓存，其余请求等待锁释放后直接读缓存。代码中使用了双重检查（DCL）模式：加锁前检查一次缓存，加锁后再检查一次，避免锁释放后重复重建缓存。

A5. ⭐⭐ 缓存穿透、击穿、雪崩分别是什么？项目里是怎么处理的？

答：

  • 穿透：请求的数据在缓存和数据库都不存在（如恶意请求不存在的短链接），每次都穿透到数据库。本项目用布隆过滤器 + 空值缓存解决。
  • 击穿：某个热点数据的缓存过期，大量并发请求同时击穿到数据库。本项目用 Redisson 分布式锁 + 双重检查解决。
  • 雪崩：大量缓存同时过期，导致大量请求涌入数据库。本项目中短链接缓存 TTL 根据有效期动态计算（永久有效的链接设置较长 TTL），分散了过期时间，同时布隆过滤器和空值缓存也提供了额外保护。

A6. ⭐⭐ 跳转时为什么需要 t_link_goto 这张路由表？

答：
t_link 表以 gid 作为分片键，ShardingSphere 路由查询时必须携带 gid 才能确定访问哪个分片。但用户访问短链接时只有 fullShortUrl，并不知道 gid 是什么。因此引入了以 fullShortUrl 为分片键的路由表 t_link_goto，专门存储 fullShortUrl → gid 的映射关系。跳转时先查路由表拿到 gid，再用 gid + fullShortUrl 查主表。这是分库分表场景下的经典”路由表”设计模式。

A7. ⭐⭐⭐ 如果布隆过滤器服务重启，数据丢失了怎么办？

答：
有两种恢复方式：

  1. 持久化重建：服务启动时从数据库全量扫描所有有效短链接，重新加载到布隆过滤器中。因为布隆过滤器只允许添加不允许删除，历史删除的数据加进去也没关系（顶多多一点误判），不影响正确性。
  2. 使用支持持久化的布隆过滤器：Redisson 的布隆过滤器底层基于 Redis Bitmap，数据存储在 Redis 中，只要 Redis 有持久化（RDB/AOF），重启后数据不丢失。

A8. ⭐⭐ UV 和 UIP 是怎么统计的？为什么用 Redis Set？

答：

  • UV（独立访客）：用 Cookie 标识用户。首次访问时生成 UUID 写入 Cookie（有效期30天），每次请求读取 Cookie 中的 uv 值，用 Redis.sadd(UV_KEY:{fullShortUrl}, uv) 添加，返回值为 1 说明是新成员（首次访问），0 说明已存在（非首次）。
  • UIP（独立 IP）：用同样的方式，将真实 IP（从 X-Forwarded-For 解析）添加到 Redis Set，判断是否首次。
  • 用 Redis Set 的原因：Set 的 sadd 操作是原子的，天然去重，且 O(1) 时间复杂度，非常适合高并发下的唯一计数场景。

模块 B：消息队列 & 异步统计

B1. ⭐⭐ 为什么统计写入要用消息队列异步处理，而不是同步写入？

答：
短链接跳转是高频操作，每次跳转需要写入 8 张统计表（access_stats、locale_stats、browser_stats、os_stats、device_stats、network_stats、access_logs、stats_today），还要调用高德地图 IP 定位 API。如果同步执行，跳转接口的响应时间会从几毫秒变成几百毫秒，用户体验极差。

使用 Redis Stream 消息队列后，跳转接口只需发一条消息（几乎无耗时），统计写入由后台消费者异步完成，跳转接口响应时间大幅降低。

B2. ⭐⭐ 为什么选 Redis Stream 而不是 RocketMQ 或 Kafka？

答：
在学习项目中选择 Redis Stream 有以下考量：

  1. 减少中间件依赖：项目已经依赖 Redis，使用 Redis Stream 不需要额外部署 MQ 中间件，降低部署复杂度。
  2. 功能足够：Redis Stream 支持消费者组、消息确认、消息持久化，满足本项目需求。
  3. 学习成本低：与 Spring Data Redis 集成简单，实现 StreamListener 即可。

生产环境对于更高吞吐量、更强可靠性要求，会选择 RocketMQ（支持事务消息、延迟消息）或 Kafka（吞吐量极高）。

B3. ⭐⭐ 消息幂等是如何实现的？为什么需要幂等？

答：
为什么需要：消费者在处理消息途中崩溃重启后，消息队列会重新投递该消息，导致统计数据被重复写入（如 PV +2 而不是 +1）。

实现方式：使用 Redis 两阶段标记：

  1. 消费开始时：setIfAbsent(idempotent:{messageId}, "0", 2min)
    • 返回 true（设置成功）→ 首次消费，继续处理
    • 返回 false（key 已存在）→ 已在消费或已完成，进入下一步判断
  2. 检查完成状态：get(idempotent:{messageId})
    • 值为 “1” → 已完成，直接跳过
    • 值为 “0” → 流程未完成（崩溃重启场景），抛异常让 MQ 重试
  3. 消费成功后：set(idempotent:{messageId}, "1", 2min) 标记完成
  4. 消费异常时：delete(idempotent:{messageId}) 删除标记，允许重试

B4. ⭐⭐⭐ 跨分组修改短链接时，为什么要用读写锁？

答：
当短链接从分组A移到分组B时，操作流程是：逻辑删除旧记录（gid=A）→ 插入新记录（gid=B）→ 更新路由表 gid。在这个过程中，如果统计消费者恰好在消费这条短链接的访问记录，它会查路由表得到 gid，然后用 gid 写统计数据。如果此时 gid 正在切换，消费者可能用旧 gid 写入统计，导致数据关联错误。

解决方案：使用 Redisson 读写锁（RReadWriteLock）：

  • 修改操作加写锁：防止统计消费者同时读取
  • 统计消费加读锁：多个统计消费可以并发，但不能和修改并发
  • 读读共享，读写互斥，写写互斥

模块 C：数据库分片

C1. ⭐⭐ 项目里是怎么做分库分表的？用了哪些分片键？

答：
使用 ShardingSphere 做水平分片，每张表分 16 个分片（_0 到 _15）：

表 分片键 原因
t_user username 按用户名查询，username 分片保证同一用户数据在同一分片
t_group username 查询某用户的分组列表，与 t_user 同键保证亲和性
t_link gid 按分组查询短链接，gid 分片使同组数据在同一分片
t_link_goto full_short_url 跳转时只有 fullShortUrl，必须以此为分片键

C2. ⭐⭐ t_link 的唯一索引为什么是 (full_short_url, del_time) 而不是 full_short_url？

答：
因为同一个短链接可以被创建、删除、再次创建，如果唯一索引只有 full_short_url，第二次创建时就会因为唯一键冲突而失败。

引入 del_time（删除时间戳）后：

  • 未删除的记录：del_time = 0，(full_short_url, 0) 唯一
  • 删除后的记录：del_time = 具体时间戳，(full_short_url, 时间戳) 也唯一
  • 再次创建时：新记录 del_time = 0，与已删除记录的时间戳不同，不冲突

这是一种巧妙的”逻辑删除 + 唯一索引”兼容方案。

C3. ⭐⭐ ShardingSphere 分片后，如何防止查询时跨分片扫描（全路由）？

答：
关键是查询 SQL 必须携带分片键。本项目中：

  • 查用户：WHERE username = ? （携带分片键，单分片查询）
  • 查分组：WHERE username = ? （携带分片键）
  • 查短链接：WHERE gid = ? AND full_short_url = ? （携带分片键 gid）
  • 查路由表：WHERE full_short_url = ? （携带分片键）

如果 SQL 中没有分片键，ShardingSphere 会对所有 16 个分片全部查询再聚合，即”全路由”，性能急剧下降。所以分片键的选择要和业务查询场景对齐。

C4. ⭐⭐⭐ 如果要查询某用户所有分组下的所有短链接，会有什么问题？

答：
t_link 以 gid 分片，如果需要按 username 查所有短链接，SQL 中没有 gid，会触发全路由扫描（16张表全扫）。

本项目的解法是：先查 t_group 拿到该用户所有 gid 列表，再用 gid IN (…) 查 t_link。ShardingSphere 对 IN 条件中的每个 gid 单独路由，只访问涉及的分片，避免全路由。这也是为什么回收站分页查询要先获取用户 gid 列表的原因。

模块 D：网关 & 认证

D1. ⭐⭐ JWT 和 Token + Redis 方案有什么区别？项目为什么用后者？

答：

  • JWT：无状态，Token 本身携带用户信息，服务端不存储，但无法主动失效（退出登录后 Token 还在有效期内可用）
  • Token + Redis：有状态，Token 是随机 UUID，真实信息存在 Redis，服务端可以主动删除 Redis 中的 Token 实现强制下线和退出登录

本项目选择 Token + Redis 方案，优点是可以精确控制 Token 有效期（每次请求可刷新TTL），退出登录立即生效，安全性更高。缺点是每次鉴权都需要查一次 Redis，增加了网络开销（但 Redis 延迟通常在 1ms 以内，可接受）。

D2. ⭐⭐ 网关是怎么把用户信息传递给下游服务的？

答：
分三步：

  1. 前端请求携带 username 和 token 两个 Header
  2. 网关的 TokenValidateGatewayFilterFactory 用 token 从 Redis Hash 中取出完整用户信息（JSON），解析出 userId 和 realName，通过 ServerHttpRequest.mutate().headers() 追加到请求 Header 中
  3. 下游 Admin 服务的 UserTransmitFilter（Servlet Filter）从 HttpServletRequest Header 中读取 username、userId、realName，封装成 UserInfoDTO 存入 ThreadLocal（UserContext），Service 层通过 UserContext.getUsername() 获取当前用户

这样做的好处是：Service 层不需要关心如何获取用户信息，统一由 Filter 层注入，代码解耦清晰。

D3. ⭐ 什么是 ThreadLocal？为什么用 ThreadLocal 存用户信息？

答：
ThreadLocal 是 Java 提供的线程本地变量，每个线程拥有独立的变量副本，线程间互不干扰。

用于存用户信息的原因：

  • Web 请求是多线程处理的，每个请求由独立线程处理
  • 将用户信息存入 ThreadLocal 后，同一请求链路中任何位置（Controller → Service → DAO）都可以直接获取，不需要通过方法参数层层传递
  • 请求结束后必须调用 UserContext.removeUser() 清除（本项目在 Filter 的 finally 块中清除），防止内存泄漏和线程池复用时的数据污染

D4. ⭐⭐ 用户限流是怎么做的？

答：
使用 UserFlowRiskControlFilter + Lua 脚本实现滑动窗口限流。核心逻辑在 Lua 脚本（user_flow_risk_control.lua）中，以用户名为 key，在指定时间窗口（timeWindow 秒）内统计请求次数，超过 maxAccessCount 则返回限流错误。

使用 Lua 脚本的原因：Redis 执行 Lua 脚本是原子操作，避免了「查询计数 → 判断 → 自增」这三步操作之间的竞态条件。配置在 application.yaml 中：时间窗口 1 秒，最大访问次数 20 次。

模块 E：分组管理 & 用户注册

E1. ⭐⭐ gid 是怎么保证全局唯一的？用了哪些手段？

答：
三层保障：

  1. 布隆过滤器（gidRegisterCachePenetrationBloomFilter）：快速判断 gid 是否已存在，O(1) 时间复杂度，不存在则可用
  2. t_group_unique 表唯一索引：在布隆过滤器之后，尝试向 t_group_unique 插入该 gid，如果抛出 DuplicateKeyException 说明已存在（布隆过滤器误判），返回 null 重试
  3. Redisson 分布式锁：创建分组时加锁（key = LOCK_GROUP_CREATE:{username}），防止并发创建时同一用户超过最大分组数限制

这是「布隆过滤器 + 数据库唯一索引」双重保障的经典模式。

E2. ⭐ 用户注册时为什么要创建一个默认分组？

答：
短链接必须归属到某个分组（gid 是必填字段），如果用户注册后没有分组就无法创建短链接。因此在注册成功后，UserServiceImpl.register() 会自动调用 GroupService.saveGroup(username, "默认分组")，确保每个用户至少有一个可用分组，提升用户体验，避免用户注册后第一步就遇到障碍。

模块 F：Spring Cloud 微服务

F1. ⭐⭐ 项目中 admin 调用 project 为什么用 OpenFeign 而不是 RestTemplate？

答：

  • RestTemplate：手动拼接 URL、手动序列化请求体、手动反序列化响应，代码冗长
  • OpenFeign：声明式 HTTP 客户端，只需定义接口 + 注解（@FeignClient、@GetMapping 等），框架自动生成实现，代码简洁，与 Spring MVC 风格一致，与 Nacos 服务发现无缝集成，支持负载均衡（lb://service-name）

本项目的 ShortLinkActualRemoteService 接口只需声明方法签名，就能自动调用 project 服务的对应接口，非常清晰。

F2. ⭐ Nacos 在项目中起什么作用？

答：
Nacos 作为服务注册与发现中心：

  • gateway、admin、project 三个服务启动时都向 Nacos 注册自己的 IP 和端口
  • 网关路由配置 uri: lb://short-link-admin 中，lb:// 前缀表示通过 Nacos 服务发现 + 负载均衡找到 admin 服务实例
  • admin 的 OpenFeign 调用 short-link-project 时，也通过 Nacos 获取 project 的真实地址
  • 如果某个服务有多个实例，Nacos + 负载均衡会自动分配请求

F3. ⭐⭐ 项目中 aggregation 模块是做什么的？

答：
aggregation 是聚合部署模块，将 admin 和 project 两个服务打包成一个 JAR 包运行，用于开发调试或资源有限的部署环境。

此时 ShortLinkActualRemoteService（OpenFeign 接口）的 url 配置从空（走 Nacos 服务发现）变为 aggregation.remote-url（直连本地），跳过 Nacos，两个服务在同一 JVM 内通过 HTTP 调用（localhost），避免了跨网络开销。这体现了微服务架构设计上对部署灵活性的考虑。

模块 G：通用 Java & Spring 基础（结合项目回答）

G1. ⭐ Spring Boot 和 Spring Cloud 有什么区别？

答：

  • Spring Boot：简化单体 Spring 应用开发，自动配置、内嵌 Tomcat、starter 依赖管理
  • Spring Cloud：基于 Spring Boot，提供微服务开发的一套解决方案，包括服务注册发现（Nacos）、网关（Gateway）、服务调用（OpenFeign）、负载均衡、配置中心等

本项目中 admin、project、gateway 各自是 Spring Boot 应用，它们通过 Spring Cloud 的 Nacos、Gateway、OpenFeign 组合成微服务系统。

G2. ⭐⭐ MyBatis-Plus 是什么？项目里怎么用的？

答：
MyBatis-Plus 是 MyBatis 的增强工具，提供了通用 CRUD 方法（BaseMapper）、条件构造器（LambdaQueryWrapper）、分页插件等，减少大量重复 SQL 编写。

本项目中：

  • Service 继承 ServiceImpl<Mapper, DO>，自动获得 insert/update/selectList 等方法
  • 使用 LambdaQueryWrapper 和 LambdaUpdateWrapper 构建类型安全的查询条件，避免字段名写错
  • 分页查询通过 IPage + 分页插件实现
  • 对于复杂 SQL（如按排序字段分页、统计聚合），使用自定义 XML Mapper

G3. ⭐⭐ 项目中用了哪些设计模式？

答：

  • 过滤器链（责任链）：请求经过 UserTransmitFilter → UserFlowRiskControlFilter → Controller，每个 Filter 只处理自己的职责
  • 模板方法：AbstractGatewayFilterFactory 定义了过滤器工厂的框架，TokenValidateGatewayFilterFactory 实现具体逻辑
  • 建造者模式：大量使用 Lombok @Builder 注解（如 ShortLinkDO.builder().gid().originUrl().build()），链式构建对象
  • 门面模式：ShortLinkActualRemoteService 作为 admin 访问 project 所有接口的统一门面
  • 策略模式（隐含）：短链接有效期类型（永久/自定义）对应不同的 Redis TTL 计算策略

G4. ⭐ @Transactional 注解用在哪里了？有没有什么注意事项？

答：
项目中 createShortLink()、updateShortLink()、register() 等方法上加了 @Transactional(rollbackFor = Exception.class)，确保短链接主表和路由表的写入要么都成功要么都回滚。

注意事项：

  1. rollbackFor = Exception.class 必须写，否则默认只对 RuntimeException 回滚，受检异常不回滚
  2. 事务方法不能在同类内部调用（会导致代理失效，事务不生效）

12.3 关键设计决策与类协调

1. 防缓存穿透（布隆过滤器）

  • RBloomFilter<String> shortUriCreateCachePenetrationBloomFilter 存储所有有效的 fullShortUrl
  • 跳转时先查布隆过滤器，不存在则直接返回 notfound，不查数据库
  • 注册时 userRegisterCachePenetrationBloomFilter 防止用户名重复查库

2. 防缓存击穿（分布式锁）

  • RLock(LOCK_GOTO_SHORT_LINK_KEY:{fullShortUrl}) 保证同一短链接缓存失效后，只有一个请求重建缓存，其余请求等锁后直接读缓存

3. 防缓存雪崩（空值缓存）

  • 对不存在/已过期的短链接，写入 GOTO_IS_NULL_SHORT_LINK_KEY:{fullShortUrl} = "-" 空值缓存（30分钟）
  • 防止大量无效请求击穿数据库

4. 统计数据一致性（读写锁）

  • updateShortLink 跨分组移动时加 写锁
  • ShortLinkStatsSaveConsumer 消费统计消息时加 读锁
  • 两者互斥，保证统计消费时 gid 不会被修改，避免统计数据写入错误分片

5. 统计幂等性（Redis 两阶段标记）

  • MessageQueueIdempotentHandler 用 setIfAbsent + value 状态（”0”/“1”）实现两阶段幂等
  • 保证消息队列重试时不会重复写入统计数据

6. admin 与 project 解耦

  • ShortLinkActualRemoteService 是 admin 模块中的 OpenFeign 客户端接口，声明了对 project 所有接口的调用
  • admin 端的 Controller 对于短链接操作几乎全部透传给 Feign，自身只处理用户和分组的本地逻辑
  • aggregation 模块用于将 admin 和 project 打包为单体部署（聚合部署模式），此时 Feign 的 url 配置为 aggregation.remote-url 直连，不走 Nacos
