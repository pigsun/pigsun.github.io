数据类型转换

隐形转换

隐式类型转换支持字节数小的类型自动转换为字节数大的类型，整数类型自动转换为小数类型，转换规则如下：【小范围转大范围】

  • byte→short(char)→int→long→float→double
  • 问题：为什么long比float大，还能转换为float呢？
  • 答：位数大小（bit size） 和 数值范围（range of values） 是两个不同的概念。
    • long: 是一个 64位 (bit) 的有符号整数类型，范围是连续的，在这个范围内的每一个整数都可以被精确表示
    • float: 是一个 32位 (bit) 的单精度浮点数类型，IEEE 754 标准，范围是极其巨大的，但是不连续的
    • 总结：Java 在设计类型转换规则时，判断“扩大”的标准是看目标类型的值域是否完全包含了源类型的值域。由于 float 的最大值远大于 long 的最大值，所以从 long 到 float 被认为是一种扩大转换。虽然这种转换会带来精度上的损失，但不会导致数值溢出到无效范围

显性转换

也叫强转换类型，牺牲精度强行进行类型转换 【大范围转小范围】

数据类型自动提升

这个特性是由 Java虚拟机规范 定义的，也是为了提高运行的效率。其他的特性还有：

  • 如果一个操作数是long型，计算结果就是long型
  • 如果一个操作数是float型，计算结果就是float型
  • 如果一个操作数是double型，计算结果就是double型
  • 优先级（double > float > long > int）

面向对象编程的三大特性。

封装

封装思想其实就是把实现细节给隐藏了，外部只需知道这个方法是什么作用，而无需关心如何实现。外界只能调用接口or使用该方法，这样将操作成员变量的权限与外界隔开。

目的：是为了保证变量的安全性，使用者不必在意具体实现细节，而只是通过外部接口即可访问类的成员，如果不进行封装，类中的实例变量可以直接查看和修改，可能给整个代码带来不好的影响，因此在编写类时一般将成员变量私有化，外部类需要同getter和setter方法来查看和设置变量。【小结：成员变量应该私有化（private），使外部只能通过getter、setter方法来查看和设置变量】

即变量私有化，操作公开化

继承

继承允许一个类（子类/派生类）获得另一个类（父类/基类/超类）的属性和方法。这是面向对象编程中实现代码复用和建立类之间层次关系的关键机制。

关键要点

  1. 关键字 extends：子类通过 class 子类名 extends 父类名 的语法来继承父类。
  2. 成员访问
    • 子类可以直接访问父类中 public 和 protected 修饰的成员（属性和方法）。
    • 子类拥有父类的 private 私有成员，但不能直接访问。需要通过父类提供的 public 或 protected 的 getter/setter 方法来间接访问。
  3. 单一继承：Java 不支持多继承（一个类不能同时继承多个类），但支持多层继承（A 继承 B，B 继承 C）。
  4. super 关键字
    • super 指代当前对象的父类。
    • super.成员变量：访问父类的成员变量。
    • super.方法名()：调用父类的方法。
    • super(...)：必须出现在子类构造方法的第一行，用于调用父类的构造方法。如果父类有有参构造方法，子类必须显式调用 super(参数)。如果父类只有无参构造方法，super() 可以省略，编译器会默认添加

多态

多态是指同一个方法调用，由于对象的实际类型不同，会产生不同的行为表现。它极大地提高了程序的扩展性和可维护性

实现多态的三个必要条件

  1. 继承 (Inheritance)：必须存在父子类关系。
  2. 重写 (Overriding)：子类必须重写父类的方法，以提供自己特定的实现。
  3. 向上转型 (Upcasting)：父类的引用变量可以指向其子类的实例。

关键要点

  1. 方法重写 (Override) vs 方法重载 (Overload)：

    • 重写 (Override)：发生在父子类之间，方法名、参数列表、返回值类型完全相同（或返回值是子类型）。使用 @Override 注解可以帮助检查。
    • 重载 (Overload)：发生在同一个类中，方法名相同，但参数列表不同。
    • 静态方法：不能被重写。父类和子类中同名的静态方法是各自独立的。

​ 3.向下转型 (Downcasting)：将父类引用强制转换回子类引用，以便调用子类特有的方法。这需要使用 instanceof 进行安全检查，以防 ClassCastException。

  1. instanceof 关键字：用于在运行时判断一个对象是否是某个类（或其子类）的实例。

总结

  • 继承是基础，它定义了类之间的“是什么”(IS-A) 关系，实现了代码复用。
  • 多态是目的，它利用继承和重写，实现了接口的通用性，让代码更加灵活和易于扩展。
  • 重写是连接继承和多态的桥梁，它让父类的接口得以在子类中展现不同的行为。

final

1. 修饰变量 (final variable)

  • 基本数据类型：变量的值不能被改变。
  • 引用数据类型：变量的引用地址不能被改变，但所指向对象的内部状态是可以修改的。

2. 修饰方法 (final method)

  • 该方法不能被子类重写（Override）。
  • 目的：保护方法的实现不被修改，通常用于核心、关键的算法或逻辑。

3. 修饰类 (final class)

  • 该类不能被继承。
  • 目的：防止类被扩展，保证类的完整性。Java 中的 String, Integer 等包装类都是 final 类。

抽象类（Abstract Class）

关键特征

  1. abstract 修饰符：定义抽象类和抽象方法都必须使用 abstract 关键字。
  2. 不能被实例化：不能直接使用 new 来创建抽象类的对象。

​ 3.可以包含普通成员：抽象类不仅可以包含抽象方法，还可以包含普通的成员变量、普通方法、构造方法等。

​ 4.强制子类实现：任何继承抽象类的非抽象子类，都必须实现其所有的抽象方法，除非子类本身也是抽象类。

​ 5.匿名内部类实现：可以创建抽象类的匿名子类来实现其抽象方法。

接口 (Interface)

接口是比抽象类更纯粹的抽象，它代表着一种“能力”或“契约”。

关键特征

  1. interface 修饰符：定义接口使用 interface 关键字。

  2. 纯抽象：在 Java 8 以前，接口中的所有方法都是 public abstract 的（这两个关键字可以省略）。

  3. 常量：接口中定义的变量默认是 public static final 的，即公共、静态、不可变的常量。

  4. default 方法 (Java 8+)：可以为接口方法提供一个默认实现，这样实现类就不强制必须重写这个方法。

  5. 实现接口：类使用 implements 关键字来实现接口。一个类可以实现多个接口，从而具备多种能力。

  6. 多态性：接口引用可以指向任何实现了该接口的类的实例。（这个特性让接口成为 Java 中实现松耦合、高扩展性架构的基石）

    为什么这很重要？

    1. 解耦合 (Decoupling)：drawShape 方法只知道它接收的是一个 Drawable，它并不关心这个对象具体是 Circle 还是 Rectangle。这就将方法的实现与具体的实现类解耦了。
    2. 可扩展性 (Extensibility)：如果未来需要添加一个新的图形，比如 Star，我们只需要让 Star 类实现 Drawable 接口，然后就可以直接将 Star 的实例传给 drawShape 方法，而无需修改 drawShape 的代码。

    接口引用指向实现类实例的特性，是接口实现“多态”的核心机制，它让我们的代码更加灵活、健壮和易于维护。

匿名内部类

匿名内部类的核心思想：当一个类的实现（或方法的重写）只使用一次，并且实现逻辑相对简单时，可以使用匿名内部类来快速、简洁地创建一个对象，而无需为其定义一个正式的、有名字的类。

优点：代码简洁，逻辑内聚，适用于一次性任务。
缺点：无法复用，不适合复杂逻辑，过多使用可能影响代码可读性。

包装类

其核心意义——桥接基本数据类型和面向对象世界

为什么需要包装类？

Java 的基本数据类型（int, double, boolean 等）不是对象，它们是直接存储在栈中的原始值。这与 Java “一切皆对象”的面向对象思想有所冲突。

  1. 面向对象的需求：有时候我们需要将一个整数作为一个对象来处理。例如，将它作为方法的参数传递给期望接收对象的方法，或者把它存储在只接受对象的集合（如 List）中。
  2. 提供工具方法：包装类提供了许多实用的静态方法，用于在字符串和基本类型之间进行转换，或者获取类型的相关信息。

Java异常机制

异常机制分为异常（Exception）和错误（Error）

异常（Exception）

运行时异常

定义：在编译阶段无法感知代码是否会出现问题，只有在运行的时候才知道会不会出错（正常情况下是不会出错的），这样的异常称为运行时异常。

特点：所有的运行时异常都直接继承自RuntimeException（RuntimeException也是继承Exception）

编译时异常

定义：在编译阶段就需要进行处理的异常（捕获异常）如果不进行处理，将无法通过编译！

特点：默认直接继承自Exception类的异常都是编译时异常

错误（Error）

错误比异常更严重，异常就是不同寻常，但不一定会导致致命的问题，而错误是致命问题，一般出现错误可能JVM就无法继续正常运行了

异常处理

程序出现异常时，默认会交给JVM来处理，JVM发现任何异常都会立即终止程序运行，并在控制台打印栈追踪信息。这是就需要编写程序手动捕获异常，使程序继续正常运行。(一旦手动捕获后,异常就不再交给JVM处理)

运行时异常在编译时可以不用捕获，但是编译时异常必须进行处理

泛型

核心目的：泛型（Generics）的核心目的是在编译期就发现类型不匹配的问题，从而避免在运行时发生 ClassCastException。

类型擦除 (Type Erasure)

这是泛型实现的关键机制。正如您所说，泛型信息在编译后会被擦除。

  • 编译前：Score<String>, Score<Integer>
  • 编译后：Score，所有 T 都被替换为它的上界（通常是 Object）。

关键点：虽然类型被擦除了，但编译器会在必要的地方自动插入类型转换代码，确保类型安全。

泛型的使用细节

  • 类上使用：public class MyClass<T> {}

  • 方法上使用：

    • 依赖于类：public T getValue() {} (这个 T 从属于类的 <T>)
    • 独立于类 (自定义)：public <E> void printElement(E element) {} (这个 <E> 是方法独有的)

  • 静态成员：静态成员属于类本身，而不是实例。在类加载时，泛型信息还未确定，因此静态变量不能使用类的泛型参数。但静态方法可以有自己的泛型参数。

  • 基本类型：泛型不能是 int, double, boolean 等基本数据类型，必须是它们的包装类 Integer, Double, Boolean。

I/O流

Java I/O 的设计大量使用了装饰者模式 (Decorator Pattern)。其核心思想是：

  1. 基础流 (Base Stream)：如 FileInputStream / FileOutputStream / FileReader / FileWriter，负责与数据源（如文件）建立连接。

  2. 装饰流 (Decorating Stream)：如 BufferedInputStream / InputStreamReader / ObjectInputStream 等，它们接收一个基础流作为构造参数，为其“装饰”上新的功能（如缓冲、字符转换、对象序列化等）。

    小结：

    除文件流外的五个流只能嵌套文件流，他们之间无法相互嵌套，所以用流的时候需要判断应该什么流嵌套文件流

这种模式使得 I/O 流的功能可以像搭积木一样灵活组合，非常强大。

基础：文件流 (File Streams)

直接操作文件的流

  • 文件字节流：FileInputStream / FileOutputStream
    • 适用场景：所有类型的文件（图片、视频、文本等）。数据以 byte 为单位读写。
    • 注意：处理文本文件时，需要考虑字符编码问题（一个中文字符可能占多个字节）。
  • 文件字符流：FileReader / FileWriter
    • 适用场景：纯文本文件。数据以 char 为单位读写，内部自动处理了字符编码转换。
    • 注意：没有 available() 方法。

转换流 (Conversion Streams) - 解决编码问题

这是 I/O 流中非常重要的一个环节，它连接了字节流和字符流。

  • InputStreamReader：将字节输入流（如 FileInputStream）转换为字符输入流（Reader）。
  • OutputStreamWriter：将字符输出流（Writer）转换为字节输出流（如 FileOutputStream）。

意义：允许我们在处理文件时，可以指定字符集（如 UTF-8, GBK），解决了乱码问题。

缓冲流 (Buffered Streams) - 提升性能

对基础流进行性能优化

  • 原理：在内存中开辟一个缓冲区。当从流中读取数据时，不是每次只读一个字节/字符，而是先将数据批量读入缓冲区，程序再从缓冲区读取。写入同理，先写入缓冲区，再批量写入目标。这大大减少了实际的 I/O 操作次数。
  • 类型：BufferedInputStream / BufferedOutputStream / BufferedReader / BufferedWriter
  • mark() 和 reset()：BufferedInputStream 和 BufferedReader 提供了这两个方法，允许你在读取流的过程中标记一个位置，之后可以调用 reset() 返回到该位置。这在解析数据时非常有用。

模块 A：短链接跳转 & 缓存设计

A1. ⭐ 短链接是怎么生成的？

答：
对原始 URL 拼接一个随机 UUID 后，使用 MurmurHash 算法计算哈希值，再转换为 Base62 编码（0-9a-zA-Z），截取前 6 位作为短链接后缀。之所以拼接 UUID 是为了每次生成不同的哈希输入，降低冲突概率。生成后检查布隆过滤器是否已存在，存在则重试，最多重试 10 次。

A2. ⭐⭐ 为什么选布隆过滤器而不是直接查数据库判断短链接是否重复？

答：
布隆过滤器是纯内存操作，查询时间复杂度 O(k)（k 为哈希函数数量），性能远超数据库查询。短链接系统中，短链接的数量可能达到亿级，每次创建都查一次数据库压力极大。布隆过滤器的缺点是有一定误判率（可能误判为存在），但不会漏判（不存在一定返回不存在），对于防重场景完全够用，误判时最多多重试一次。

A3. ⭐⭐ 短链接跳转时是如何防止缓存穿透的？

答：
使用了两层防护：

  1. 布隆过滤器：存储所有有效的完整短链接（fullShortUrl），跳转时先查布隆过滤器，如果不存在直接返回 404，不会访问数据库。
  2. 空值缓存：对于布隆过滤器误判通过但数据库中不存在（或已过期）的短链接，在 Redis 中写入一个空值缓存（key = GOTO_IS_NULL:{fullShortUrl}, value = “-“），TTL 30 分钟，后续请求直接返回 404，不再查库。

A4. ⭐⭐ 短链接跳转时是如何防止缓存击穿的？

答：
使用 Redisson 分布式锁。当缓存中没有该短链接时，多个并发请求同时到达，只有第一个请求能获取到锁并查询数据库重建缓存，其余请求等待锁释放后直接读缓存。代码中使用了双重检查（DCL）模式：加锁前检查一次缓存，加锁后再检查一次，避免锁释放后重复重建缓存。

A5. ⭐⭐ 缓存穿透、击穿、雪崩分别是什么？项目里是怎么处理的？

答：

  • 穿透：请求的数据在缓存和数据库都不存在（如恶意请求不存在的短链接），每次都穿透到数据库。本项目用布隆过滤器 + 空值缓存解决。
  • 击穿：某个热点数据的缓存过期，大量并发请求同时击穿到数据库。本项目用 Redisson 分布式锁 + 双重检查解决。
  • 雪崩：大量缓存同时过期，导致大量请求涌入数据库。本项目中短链接缓存 TTL 根据有效期动态计算（永久有效的链接设置较长 TTL），分散了过期时间，同时布隆过滤器和空值缓存也提供了额外保护。

A6. ⭐⭐ 跳转时为什么需要 t_link_goto 这张路由表？

答：
t_link 表以 gid 作为分片键，ShardingSphere 路由查询时必须携带 gid 才能确定访问哪个分片。但用户访问短链接时只有 fullShortUrl，并不知道 gid 是什么。因此引入了以 fullShortUrl 为分片键的路由表 t_link_goto，专门存储 fullShortUrl → gid 的映射关系。跳转时先查路由表拿到 gid，再用 gid + fullShortUrl 查主表。这是分库分表场景下的经典”路由表”设计模式。

A7. ⭐⭐⭐ 如果布隆过滤器服务重启，数据丢失了怎么办？

答：
有两种恢复方式：

  1. 持久化重建：服务启动时从数据库全量扫描所有有效短链接，重新加载到布隆过滤器中。因为布隆过滤器只允许添加不允许删除，历史删除的数据加进去也没关系（顶多多一点误判），不影响正确性。
  2. 使用支持持久化的布隆过滤器：Redisson 的布隆过滤器底层基于 Redis Bitmap，数据存储在 Redis 中，只要 Redis 有持久化（RDB/AOF），重启后数据不丢失。

A8. ⭐⭐ UV 和 UIP 是怎么统计的？为什么用 Redis Set？

答：

  • UV（独立访客）：用 Cookie 标识用户。首次访问时生成 UUID 写入 Cookie（有效期30天），每次请求读取 Cookie 中的 uv 值，用 Redis.sadd(UV_KEY:{fullShortUrl}, uv) 添加，返回值为 1 说明是新成员（首次访问），0 说明已存在（非首次）。
  • UIP（独立 IP）：用同样的方式，将真实 IP（从 X-Forwarded-For 解析）添加到 Redis Set，判断是否首次。
  • 用 Redis Set 的原因：Set 的 sadd 操作是原子的，天然去重，且 O(1) 时间复杂度，非常适合高并发下的唯一计数场景。

模块 B：消息队列 & 异步统计

B1. ⭐⭐ 为什么统计写入要用消息队列异步处理，而不是同步写入？

答：
短链接跳转是高频操作，每次跳转需要写入 8 张统计表（access_stats、locale_stats、browser_stats、os_stats、device_stats、network_stats、access_logs、stats_today），还要调用高德地图 IP 定位 API。如果同步执行，跳转接口的响应时间会从几毫秒变成几百毫秒，用户体验极差。

使用 Redis Stream 消息队列后，跳转接口只需发一条消息（几乎无耗时），统计写入由后台消费者异步完成，跳转接口响应时间大幅降低。

B2. ⭐⭐ 为什么选 Redis Stream 而不是 RocketMQ 或 Kafka？

答：
在学习项目中选择 Redis Stream 有以下考量：

  1. 减少中间件依赖：项目已经依赖 Redis，使用 Redis Stream 不需要额外部署 MQ 中间件，降低部署复杂度。
  2. 功能足够：Redis Stream 支持消费者组、消息确认、消息持久化，满足本项目需求。
  3. 学习成本低：与 Spring Data Redis 集成简单，实现 StreamListener 即可。

生产环境对于更高吞吐量、更强可靠性要求，会选择 RocketMQ（支持事务消息、延迟消息）或 Kafka（吞吐量极高）。

B3. ⭐⭐ 消息幂等是如何实现的？为什么需要幂等？

答：
为什么需要：消费者在处理消息途中崩溃重启后，消息队列会重新投递该消息，导致统计数据被重复写入（如 PV +2 而不是 +1）。

实现方式：使用 Redis 两阶段标记：

  1. 消费开始时：setIfAbsent(idempotent:{messageId}, "0", 2min)
    • 返回 true（设置成功）→ 首次消费，继续处理
    • 返回 false（key 已存在）→ 已在消费或已完成，进入下一步判断
  2. 检查完成状态：get(idempotent:{messageId})
    • 值为 “1” → 已完成，直接跳过
    • 值为 “0” → 流程未完成（崩溃重启场景），抛异常让 MQ 重试
  3. 消费成功后：set(idempotent:{messageId}, "1", 2min) 标记完成
  4. 消费异常时：delete(idempotent:{messageId}) 删除标记，允许重试

B4. ⭐⭐⭐ 跨分组修改短链接时，为什么要用读写锁？

答：
当短链接从分组A移到分组B时，操作流程是：逻辑删除旧记录（gid=A）→ 插入新记录（gid=B）→ 更新路由表 gid。在这个过程中，如果统计消费者恰好在消费这条短链接的访问记录，它会查路由表得到 gid，然后用 gid 写统计数据。如果此时 gid 正在切换，消费者可能用旧 gid 写入统计，导致数据关联错误。

解决方案：使用 Redisson 读写锁（RReadWriteLock）：

  • 修改操作加写锁：防止统计消费者同时读取
  • 统计消费加读锁：多个统计消费可以并发，但不能和修改并发
  • 读读共享，读写互斥，写写互斥

模块 C：数据库分片

C1. ⭐⭐ 项目里是怎么做分库分表的？用了哪些分片键？

答：
使用 ShardingSphere 做水平分片，每张表分 16 个分片（_0 到 _15）：

表 分片键 原因
t_user username 按用户名查询，username 分片保证同一用户数据在同一分片
t_group username 查询某用户的分组列表，与 t_user 同键保证亲和性
t_link gid 按分组查询短链接，gid 分片使同组数据在同一分片
t_link_goto full_short_url 跳转时只有 fullShortUrl，必须以此为分片键

C2. ⭐⭐ t_link 的唯一索引为什么是 (full_short_url, del_time) 而不是 full_short_url？

答：
因为同一个短链接可以被创建、删除、再次创建，如果唯一索引只有 full_short_url，第二次创建时就会因为唯一键冲突而失败。

引入 del_time（删除时间戳）后：

  • 未删除的记录：del_time = 0，(full_short_url, 0) 唯一
  • 删除后的记录：del_time = 具体时间戳，(full_short_url, 时间戳) 也唯一
  • 再次创建时：新记录 del_time = 0，与已删除记录的时间戳不同，不冲突

这是一种巧妙的”逻辑删除 + 唯一索引”兼容方案。

C3. ⭐⭐ ShardingSphere 分片后，如何防止查询时跨分片扫描（全路由）？

答：
关键是查询 SQL 必须携带分片键。本项目中：

  • 查用户：WHERE username = ? （携带分片键，单分片查询）
  • 查分组：WHERE username = ? （携带分片键）
  • 查短链接：WHERE gid = ? AND full_short_url = ? （携带分片键 gid）
  • 查路由表：WHERE full_short_url = ? （携带分片键）

如果 SQL 中没有分片键，ShardingSphere 会对所有 16 个分片全部查询再聚合，即”全路由”，性能急剧下降。所以分片键的选择要和业务查询场景对齐。

C4. ⭐⭐⭐ 如果要查询某用户所有分组下的所有短链接，会有什么问题？

答：
t_link 以 gid 分片，如果需要按 username 查所有短链接，SQL 中没有 gid，会触发全路由扫描（16张表全扫）。

本项目的解法是：先查 t_group 拿到该用户所有 gid 列表，再用 gid IN (…) 查 t_link。ShardingSphere 对 IN 条件中的每个 gid 单独路由，只访问涉及的分片，避免全路由。这也是为什么回收站分页查询要先获取用户 gid 列表的原因。

模块 D：网关 & 认证

D1. ⭐⭐ JWT 和 Token + Redis 方案有什么区别？项目为什么用后者？

答：

  • JWT：无状态，Token 本身携带用户信息，服务端不存储，但无法主动失效（退出登录后 Token 还在有效期内可用）
  • Token + Redis：有状态，Token 是随机 UUID，真实信息存在 Redis，服务端可以主动删除 Redis 中的 Token 实现强制下线和退出登录

本项目选择 Token + Redis 方案，优点是可以精确控制 Token 有效期（每次请求可刷新TTL），退出登录立即生效，安全性更高。缺点是每次鉴权都需要查一次 Redis，增加了网络开销（但 Redis 延迟通常在 1ms 以内，可接受）。

D2. ⭐⭐ 网关是怎么把用户信息传递给下游服务的？

答：
分三步：

  1. 前端请求携带 username 和 token 两个 Header
  2. 网关的 TokenValidateGatewayFilterFactory 用 token 从 Redis Hash 中取出完整用户信息（JSON），解析出 userId 和 realName，通过 ServerHttpRequest.mutate().headers() 追加到请求 Header 中
  3. 下游 Admin 服务的 UserTransmitFilter（Servlet Filter）从 HttpServletRequest Header 中读取 username、userId、realName，封装成 UserInfoDTO 存入 ThreadLocal（UserContext），Service 层通过 UserContext.getUsername() 获取当前用户

这样做的好处是：Service 层不需要关心如何获取用户信息，统一由 Filter 层注入，代码解耦清晰。

D3. ⭐ 什么是 ThreadLocal？为什么用 ThreadLocal 存用户信息？

答：
ThreadLocal 是 Java 提供的线程本地变量，每个线程拥有独立的变量副本，线程间互不干扰。

用于存用户信息的原因：

  • Web 请求是多线程处理的，每个请求由独立线程处理
  • 将用户信息存入 ThreadLocal 后，同一请求链路中任何位置（Controller → Service → DAO）都可以直接获取，不需要通过方法参数层层传递
  • 请求结束后必须调用 UserContext.removeUser() 清除（本项目在 Filter 的 finally 块中清除），防止内存泄漏和线程池复用时的数据污染

D4. ⭐⭐ 用户限流是怎么做的？

答：
使用 UserFlowRiskControlFilter + Lua 脚本实现滑动窗口限流。核心逻辑在 Lua 脚本（user_flow_risk_control.lua）中，以用户名为 key，在指定时间窗口（timeWindow 秒）内统计请求次数，超过 maxAccessCount 则返回限流错误。

使用 Lua 脚本的原因：Redis 执行 Lua 脚本是原子操作，避免了「查询计数 → 判断 → 自增」这三步操作之间的竞态条件。配置在 application.yaml 中：时间窗口 1 秒，最大访问次数 20 次。

模块 E：分组管理 & 用户注册

E1. ⭐⭐ gid 是怎么保证全局唯一的？用了哪些手段？

答：
三层保障：

  1. 布隆过滤器（gidRegisterCachePenetrationBloomFilter）：快速判断 gid 是否已存在，O(1) 时间复杂度，不存在则可用
  2. t_group_unique 表唯一索引：在布隆过滤器之后，尝试向 t_group_unique 插入该 gid，如果抛出 DuplicateKeyException 说明已存在（布隆过滤器误判），返回 null 重试
  3. Redisson 分布式锁：创建分组时加锁（key = LOCK_GROUP_CREATE:{username}），防止并发创建时同一用户超过最大分组数限制

这是「布隆过滤器 + 数据库唯一索引」双重保障的经典模式。

E2. ⭐ 用户注册时为什么要创建一个默认分组？

答：
短链接必须归属到某个分组（gid 是必填字段），如果用户注册后没有分组就无法创建短链接。因此在注册成功后，UserServiceImpl.register() 会自动调用 GroupService.saveGroup(username, "默认分组")，确保每个用户至少有一个可用分组，提升用户体验，避免用户注册后第一步就遇到障碍。

模块 F：Spring Cloud 微服务

F1. ⭐⭐ 项目中 admin 调用 project 为什么用 OpenFeign 而不是 RestTemplate？

答：

  • RestTemplate：手动拼接 URL、手动序列化请求体、手动反序列化响应，代码冗长
  • OpenFeign：声明式 HTTP 客户端，只需定义接口 + 注解（@FeignClient、@GetMapping 等），框架自动生成实现，代码简洁，与 Spring MVC 风格一致，与 Nacos 服务发现无缝集成，支持负载均衡（lb://service-name）

本项目的 ShortLinkActualRemoteService 接口只需声明方法签名，就能自动调用 project 服务的对应接口，非常清晰。

F2. ⭐ Nacos 在项目中起什么作用？

答：
Nacos 作为服务注册与发现中心：

  • gateway、admin、project 三个服务启动时都向 Nacos 注册自己的 IP 和端口
  • 网关路由配置 uri: lb://short-link-admin 中，lb:// 前缀表示通过 Nacos 服务发现 + 负载均衡找到 admin 服务实例
  • admin 的 OpenFeign 调用 short-link-project 时，也通过 Nacos 获取 project 的真实地址
  • 如果某个服务有多个实例，Nacos + 负载均衡会自动分配请求

F3. ⭐⭐ 项目中 aggregation 模块是做什么的？

答：
aggregation 是聚合部署模块，将 admin 和 project 两个服务打包成一个 JAR 包运行，用于开发调试或资源有限的部署环境。

此时 ShortLinkActualRemoteService（OpenFeign 接口）的 url 配置从空（走 Nacos 服务发现）变为 aggregation.remote-url（直连本地），跳过 Nacos，两个服务在同一 JVM 内通过 HTTP 调用（localhost），避免了跨网络开销。这体现了微服务架构设计上对部署灵活性的考虑。

模块 G：通用 Java & Spring 基础（结合项目回答）

G1. ⭐ Spring Boot 和 Spring Cloud 有什么区别？

答：

  • Spring Boot：简化单体 Spring 应用开发，自动配置、内嵌 Tomcat、starter 依赖管理
  • Spring Cloud：基于 Spring Boot，提供微服务开发的一套解决方案，包括服务注册发现（Nacos）、网关（Gateway）、服务调用（OpenFeign）、负载均衡、配置中心等

本项目中 admin、project、gateway 各自是 Spring Boot 应用，它们通过 Spring Cloud 的 Nacos、Gateway、OpenFeign 组合成微服务系统。

G2. ⭐⭐ MyBatis-Plus 是什么？项目里怎么用的？

答：
MyBatis-Plus 是 MyBatis 的增强工具，提供了通用 CRUD 方法（BaseMapper）、条件构造器（LambdaQueryWrapper）、分页插件等，减少大量重复 SQL 编写。

本项目中：

  • Service 继承 ServiceImpl<Mapper, DO>，自动获得 insert/update/selectList 等方法
  • 使用 LambdaQueryWrapper 和 LambdaUpdateWrapper 构建类型安全的查询条件，避免字段名写错
  • 分页查询通过 IPage + 分页插件实现
  • 对于复杂 SQL（如按排序字段分页、统计聚合），使用自定义 XML Mapper

G3. ⭐⭐ 项目中用了哪些设计模式？

答：

  • 过滤器链（责任链）：请求经过 UserTransmitFilter → UserFlowRiskControlFilter → Controller，每个 Filter 只处理自己的职责
  • 模板方法：AbstractGatewayFilterFactory 定义了过滤器工厂的框架，TokenValidateGatewayFilterFactory 实现具体逻辑
  • 建造者模式：大量使用 Lombok @Builder 注解（如 ShortLinkDO.builder().gid().originUrl().build()），链式构建对象
  • 门面模式：ShortLinkActualRemoteService 作为 admin 访问 project 所有接口的统一门面
  • 策略模式（隐含）：短链接有效期类型（永久/自定义）对应不同的 Redis TTL 计算策略

G4. ⭐ @Transactional 注解用在哪里了？有没有什么注意事项？

答：
项目中 createShortLink()、updateShortLink()、register() 等方法上加了 @Transactional(rollbackFor = Exception.class)，确保短链接主表和路由表的写入要么都成功要么都回滚。

注意事项：

  1. rollbackFor = Exception.class 必须写，否则默认只对 RuntimeException 回滚，受检异常不回滚
  2. 事务方法不能在同类内部调用（会导致代理失效，事务不生效）

  • RLock(LOCK_GOTO_SHORT_LINK_KEY:{fullShortUrl}) 保证同一短链接缓存失效后，只有一个请求重建缓存，其余请求等锁后直接读缓存

  • 对不存在/已过期的短链接，写入 GOTO_IS_NULL_SHORT_LINK_KEY:{fullShortUrl} = "-" 空值缓存（30分钟）

4. 统计数据一致性（读写锁）

  • updateShortLink 跨分组移动时加 写锁
  • ShortLinkStatsSaveConsumer 消费统计消息时加 读锁
  • 两者互斥，保证统计消费时 gid 不会被修改，避免统计数据写入错误分片

5. 统计幂等性（Redis 两阶段标记）

  • MessageQueueIdempotentHandler 用 setIfAbsent + value 状态（”0”/“1”）实现两阶段幂等

面试官可能追问

Q1: 为什么要在 Gateway 层做 Token 验证，而不是在每个服务内部做？

A:

  • 统一鉴权：所有请求都经过同一个入口，避免重复代码

  • 安全性：非法请求在网关层就被拦截，不会浪费下游服务资源

  • 性能：网关可以快速拒绝非法请求，减少后端压力

  • 灵活性：修改鉴权逻辑只需改网关配置，不需要改所有服务

Q2: realName 为什么要 URL 编码？

A: 因为 realName 可能包含中文字符，HTTP Header 中不能直接传输非 ASCII 字符，需要 URL 编码。下游服务收到后再解码。

Q3: 如果 Redis 宕机了怎么办？

A: 网关会无法验证 Token，所有请求都会返回 401。这是一个单点故障。生产环境应该使用 Redis Sentinel 或 Redis Cluster 保证高可用。

Q4: 白名单为什么要硬编码注册接口？

A：因为 YAML 白名单的匹配逻辑是 requestPath.startsWith(whiteItem)，只能判断路径，无法区分 HTTP 方法。/api/short-link/admin/v1/user 这个路径上同时存在注册（POST，不需要 Token）和修改用户（PUT，需要 Token）两个接口，如果把路径加入 YAML 白名单，PUT 修改接口也会被放行，产生安全漏洞。所以注册接口单独在 Java 代码里用 path == “/api/…/user” && method == “POST” 做精确匹配，同时限定路径和 HTTP 方法，只放行注册，不影响同路径的其他接口。

/api/short-link/admin/v1/user 这个路径上有两个语义完全不同的接口：

HTTP 方法 接口 是否需要 Token
POST 注册新用户 不需要（还没登录）
PUT 修改用户信息 需要（已登录用户才能改）

❓ 面试官可能追问

Q1: UserTransmitFilter 的 finally 块为什么一定要 removeUser？

A: Admin 服务部署在 内置Tomcat 中，Tomcat 使用线程池处理请求，请求结束后线程会被归还到线程池复用。如果不清理 ThreadLocal，下一个用不同用户身份发来的请求被分配到这个线程时，UserContext.getUsername() 会取到上一个请求的用户名，造成越权访问的安全漏洞，同时也会导致内存泄漏。

Q2: 限流为什么不用 @SentinelResource 而用自定义 Lua 脚本？

A: 两者维度不同：

  • @SentinelResource 是接口维度限流，限制某个接口的总 QPS

  • Lua 脚本是用户维度限流，限制某个用户的请求频率

本项目两种方式都用了：project 模块的创建短链接接口用了 @SentinelResource 做接口级限流；admin 模块用 Lua 脚本做用户级限流，防止单用户恶意刷接口。

Q3: 登录时为什么检查到已登录就复用 token 而不重新生成？

A: 避免同一个用户重复登录时 Redis 中积累多个 token。如果每次登录都生成新 token，旧 token 在 TTL 到期前依然有效，相当于同时存在多个有效会话，不安全。复用已有 token 并刷新 TTL，既保持了登录态，又保证了同一时刻只有一个有效 token。

Q4: lock.lock() 和 lock.tryLock() 有什么区别？为什么注册用 tryLock，创建分组用 lock？

A:

  • tryLock()：尝试获取锁，获取不到立即返回 false，不阻塞

  • lock()：获取不到则一直阻塞等待直到获取成功

注册场景：两个请求同时注册同一用户名，后来的请求 tryLock 失败 → 直接告诉用户「用户名已存在」，逻辑上正确且响应快。

创建分组场景：用 lock () 是因为需要串行化同一用户的分组创建请求，确保分组数量检查和插入是原子的。如果用 tryLock 失败直接报错，用户体验差（用户只是想创建分组，不是并发争抢唯一资源）；而且这里的并发场景相对少，阻塞等一下是可以接受的。

Q: 统计数据是怎么异步消费的？

A（标准答案）：

> 跳转接口不直接写统计表，而是构建一个统计记录 DTO（包含 fullShortUrl、UV、UIP、OS、浏览器等信息），然后同步发送到 Redis Stream 消息队列。跳转接口立即返回 302 重定向，响应时间不受统计写入影响。

>

> 后台消费者线程监听 Redis Stream，消息到达时触发 onMessage 方法。消费者先做幂等检查（用 Redis 两阶段标记防止重复消费），然后加读锁查路由表获取 gid，最后写入 8 张统计表（聚合统计、地区统计、OS/浏览器/设备/网络统计、访问日志、今日统计）。

>

> 关键点有三个：

> 1. 幂等处理：用 Redis 的 setIfAbsent 和 get 实现两阶段标记，防止消费者重启导致消息重复消费

> 2. 读写锁：消费者加读锁，修改操作加写锁，保证跨分组修改时消费者查到的 gid 和最终写入的 gid 一致

> 3. ON DUPLICATE KEY UPDATE：同一短链接同一小时可能有多个访问，用这个 SQL 语法避免复杂的”先查后插或更新”逻辑

面试官可能追问

Q1: 为什么跳转接口要用 12 步这么复杂的流程？

A: 短链接跳转是高频操作（可能每秒百万级），如果缓存失效或被击穿，大量请求会涌入数据库导致雪崩。12 步流程是分层防护：

  • 第一层（Redis）：99% 请求秒级响应

  • 第二层（布隆过滤器）：防穿透，恶意请求不查库

  • 第三层（空值缓存）：已过期链接不重复查库

  • 第四层（分布式锁）：防击穿，缓存失效时只有一个请求重建

Q2: 为什么要用 t_link_goto 路由表？

A: t_link 以 gid 分片，跳转时只有 fullShortUrl 不知道 gid。如果直接查 t_link，ShardingSphere 无法定位分片，会全表扫描 16 张表（全路由），性能极差。路由表以 fullShortUrl 分片，先查出 gid，再用 gid + fullShortUrl 精确查 t_link，只访问一张表。

Q3: 消费者为什么要加读锁？

A: 防止修改操作跨分组移动时，消费者用旧 gid 写统计。读写锁保证：修改时加写锁 → 消费者必须等待 → 修改完成后消费者才能继续 → 消费者拿到的 gid 一定是最新的。

Q4: 为什么用 Redis Stream 而不是同步写统计表？

A: 跳转接口需要快速响应（目标 P99 < 50ms）。如果同步写 8 张统计表 + 调高德地图 API，响应时间会变成几百毫秒。用 MQ 异步处理，跳转接口只需发一条消息（几毫秒），统计写入由后台消费者完成，不阻塞用户请求。

还有一个好的方案先写 MySQL，通过 Binlog，异步更新 Redis，，Canal + MQ

模块 A：短链接跳转 & 缓存设计

A1. ⭐ 短链接是怎么生成的？

答：
对原始 URL 拼接一个随机 UUID 后，使用 MurmurHash 算法计算哈希值，再转换为 Base62 编码（0-9a-zA-Z），截取前 6 位作为短链接后缀。拼接 UUID 是为了每次生成不同的哈希输入，降低冲突概率。生成后检查布隆过滤器是否已存在，存在则重试，最多重试 10 次。

A2. ⭐⭐ 为什么选布隆过滤器而不是直接查数据库判断短链接是否重复？

答：
布隆过滤器是纯内存操作，查询时间复杂度 O(k)（k 为哈希函数数量），性能远超数据库查询。短链接系统中短链接数量可能达亿级，每次创建都查数据库压力极大。布隆过滤器的缺点是有一定误判率（可能误判为存在），但不会漏判（不存在一定返回不存在），对防重场景完全够用，误判时最多多重试一次。

A3. ⭐⭐ 短链接跳转时是如何防止缓存穿透的？

答：
使用两层防护：

  1. 布隆过滤器：存储所有有效的完整短链接（fullShortUrl），跳转时先查布隆过滤器，不存在直接返回 404，不访问数据库。
  2. 空值缓存：对于布隆过滤器误判通过但数据库中不存在（或已过期）的短链接，在 Redis 中写入空值缓存（key = GOTO_IS_NULL:{fullShortUrl}, value = "-"），TTL 30 分钟，后续请求直接返回 404，不再查库。

A4. ⭐⭐ 短链接跳转时是如何防止缓存击穿的？

答：
使用 Redisson 分布式锁。当缓存中没有该短链接时，多个并发请求同时到达，只有第一个请求能获取到锁并查询数据库重建缓存，其余请求等待锁释放后直接读缓存。代码中使用双重检查（DCL）模式：加锁前检查一次缓存，加锁后再检查一次，避免锁释放后重复重建缓存。

A5. ⭐⭐ 缓存穿透、击穿、雪崩分别是什么？项目里是怎么处理的？

答：

  • 穿透：请求的数据在缓存和数据库都不存在（如恶意请求不存在的短链接），每次都穿透到数据库。本项目用布隆过滤器 + 空值缓存解决。
  • 击穿：热点数据的缓存过期，大量并发请求同时击穿到数据库。本项目用 Redisson 分布式锁 + 双重检查解决。
  • 雪崩：大量缓存同时过期，导致大量请求涌入数据库。本项目中短链接缓存 TTL 根据有效期动态计算，分散过期时间，布隆过滤器和空值缓存也提供额外保护。

A6. ⭐⭐ 跳转时为什么需要 t_link_goto 这张路由表？

答：
t_link 表以 gid 作为分片键，ShardingSphere 路由查询时必须携带 gid 才能确定访问哪个分片。但用户访问短链接时只有 fullShortUrl，并不知道 gid。因此引入以 fullShortUrl 为分片键的路由表 t_link_goto，专门存储 fullShortUrl → gid 的映射。跳转时先查路由表拿到 gid，再用 gid + fullShortUrl 查主表。这是分库分表场景下经典的「路由表」设计模式。

A7. ⭐⭐⭐ 如果布隆过滤器服务重启，数据丢失了怎么办？

答：
有两种恢复方式：

  1. 持久化重建：服务启动时从数据库全量扫描所有有效短链接，重新加载到布隆过滤器中。因为布隆过滤器只允许添加不允许删除，历史删除的数据加进去也没关系（顶多多一点误判），不影响正确性。
  2. 使用支持持久化的布隆过滤器：Redisson 的布隆过滤器底层基于 Redis Bitmap，数据存储在 Redis 中，只要 Redis 开启持久化（RDB/AOF），重启后数据不丢失。

A8. ⭐⭐ UV 和 UIP 是怎么统计的？为什么用 Redis Set？

答：

  • UV（独立访客）：用 Cookie 标识用户。首次访问时生成 UUID 写入 Cookie（有效期30天），每次请求读取 Cookie 中的 uv 值，用 Redis.sadd(UV_KEY:{fullShortUrl}, uv) 添加，返回值为 1 说明是新成员（首次访问），0 说明已存在（非首次）。
  • UIP（独立 IP）：将真实 IP（从 X-Forwarded-For 解析）添加到 Redis Set，判断是否首次访问 IP。
  • 用 Redis Set 的原因：sadd 操作是原子的，天然去重，O(1) 时间复杂度，非常适合高并发下的唯一计数场景。

模块 B：消息队列 & 异步统计

B1. ⭐⭐ 为什么统计写入要用消息队列异步处理，而不是同步写入？

答：
短链接跳转是高频操作，每次跳转需要写入 8 张统计表，还要调用高德地图 IP 定位 API。如果同步执行，跳转接口的响应时间会从几毫秒变成几百毫秒，用户体验极差。

使用 Redis Stream 消息队列后，跳转接口只需发一条消息（几乎无耗时），统计写入由后台消费者异步完成，跳转接口响应时间大幅降低（约60%）。

B2. ⭐⭐ 为什么选 Redis Stream 而不是 RocketMQ 或 Kafka？

答：

  1. 减少中间件依赖：项目已依赖 Redis，使用 Redis Stream 不需要额外部署 MQ 中间件，降低部署复杂度。
  2. 功能足够：Redis Stream 支持消费者组、消息确认、消息持久化，满足本项目需求。
  3. 学习成本低：与 Spring Data Redis 集成简单，实现 StreamListener 即可。

生产环境更高吞吐量、更强可靠性场景会选择 RocketMQ（支持事务消息、延迟消息）或 Kafka（超高吞吐量）。

B3. ⭐⭐ 消息幂等是如何实现的？为什么需要幂等？

答：
为什么需要：消费者在处理消息途中崩溃重启后，消息队列会重新投递该消息，导致统计数据被重复写入（如 PV +2 而不是 +1）。

两阶段标记实现：

  1. 消费开始：setIfAbsent(idempotent:{messageId}, "0", 2min)
    • 返回 true → 首次消费，继续处理
    • 返回 false → key 已存在，进入下一步判断
  2. 检查完成状态：get(idempotent:{messageId})
    • 值为 "1" → 已完成，直接跳过
    • 值为 "0" → 流程未完成（崩溃重启场景），抛异常让 MQ 重试
  3. 消费成功后：set(idempotent:{messageId}, "1", 2min) 标记完成
  4. 消费异常时：delete(idempotent:{messageId}) 删除标记，允许重试

B4. ⭐⭐⭐ 跨分组修改短链接时，为什么要用读写锁？

答：
当短链接从分组A移到分组B时：逻辑删除旧记录（gid=A）→ 插入新记录（gid=B）→ 更新路由表 gid。若此时统计消费者恰好在消费这条链接的访问记录，它会查路由表得到 gid，再用 gid 写统计。若 gid 正在切换，消费者可能用旧 gid 写入统计，导致数据关联错误。

解决方案：Redisson 读写锁（RReadWriteLock）：

  • 修改操作加写锁：阻塞统计消费者同时读取
  • 统计消费加读锁：多个统计消费可并发，但不能和修改并发
  • 读读共享，读写互斥，写写互斥

B5. ⭐⭐ 统计监控数据时 RocketMQ 为什么要用同步发送？既然捕获时只打了日志，为什么不能异步发送来提高响应速度？

答：
本项目实际使用的是 Redis Stream 而非 RocketMQ，但同步发送的原因是一致的：

统计消费实际做了很多复杂处理：

  1. 写入 8 张统计表（access_stats、locale_stats、os_stats、browser_stats、device_stats、network_stats、access_logs、stats_today）
  2. 调用高德地图 API 进行 IP 定位
  3. 累加主表 total_pv/total_uv/total_uip
  4. 处理消息幂等（两阶段标记）和读写锁

用同步发送的原因：

  • 同步发送确保消息一定写入 Redis Stream，不会丢失统计数据
  • 异步发送如果失败（网络抖动、Redis 短暂不可用），消息丢失则统计数据永久缺失
  • 发送消息耗时约 1-2ms，相比查询数据库的几十毫秒，不是性能瓶颈

如要进一步优化：可用异步发送 + 本地重试队列（发送失败存入本地队列，定时重试），兼顾性能与可靠性。

模块 C：数据库分片

C1. ⭐⭐ 项目里是怎么做分库分表的？用了哪些分片键？

答：
使用 ShardingSphere 做水平分片，每张表分 16 个分片（_0 到 _15）：

表 分片键 原因

t_group username 查询某用户的分组列表，与 t_user 同键保证亲和性
t_link gid 按分组查询短链接，同组数据在同一分片
t_link_goto full_short_url 跳转时只有 fullShortUrl，必须以此为分片键

C2. ⭐⭐ t_link 的唯一索引为什么是 (full_short_url, del_time) 而不是 full_short_url？

答：
同一个短链接可以被创建、删除、再次创建。如果唯一索引只有 full_short_url，第二次创建时会唯一键冲突而失败。

引入 del_time（删除时间戳）后：

  • 未删除记录：del_time = 0，(full_short_url, 0) 唯一
  • 删除后记录：del_time = 具体时间戳，(full_short_url, 时间戳) 唯一
  • 再次创建：新记录 del_time = 0，与已删除记录时间戳不同，不冲突

这是「逻辑删除 + 唯一索引」兼容的经典方案。

C3. ⭐⭐ ShardingSphere 分片后，如何防止查询时跨分片扫描（全路由）？

答：
查询 SQL 必须携带分片键：

  • 查用户：WHERE username = ? → 单分片查询
  • 查分组：WHERE username = ? → 单分片查询
  • 查短链接：WHERE gid = ? AND full_short_url = ? → 单分片查询
  • 查路由表：WHERE full_short_url = ? → 单分片查询

若 SQL 中没有分片键，ShardingSphere 会对所有 16 个分片全部查询再聚合（「全路由」），性能急剧下降。分片键选择必须和业务查询场景对齐。

C4. ⭐⭐⭐ 如果要查询某用户所有分组下的所有短链接，会有什么问题？

答：
t_link 以 gid 分片，按 username 查所有短链接时 SQL 中没有 gid，会触发全路由扫描（16张表全扫）。

本项目解法：先查 t_group 拿到该用户所有 gid 列表，再用 gid IN (...) 查 t_link。ShardingSphere 对 IN 条件中每个 gid 单独路由，只访问涉及的分片，避免全路由。这也是为什么回收站分页查询要先获取用户 gid 列表的原因。

模块 D：网关 & 认证

D1. ⭐⭐ JWT 和 Token + Redis 方案有什么区别？项目为什么用后者？

答：

  • JWT：无状态，Token 本身携带用户信息，服务端不存储，但无法主动失效（退出登录后 Token 在有效期内仍可用）
  • Token + Redis：有状态，Token 是随机 UUID，真实信息存在 Redis，服务端可主动删除 Token 实现强制下线

本项目选择 Token + Redis，优点是退出登录立即生效，可精确控制 Token 有效期（每次请求刷新TTL）。缺点是每次鉴权需查一次 Redis，但 Redis 延迟通常在 1ms 以内，可接受。

D2. ⭐⭐ 网关是怎么把用户信息传递给下游服务的？

答：
三步传递：

  1. 前端请求携带 username 和 token 两个 Header
  2. 网关 TokenValidateGatewayFilterFactory 用 token 从 Redis Hash 取出用户信息 JSON，解析出 userId、realName，通过 ServerHttpRequest.mutate().headers() 追加到请求 Header
  3. 下游 Admin 服务的 UserTransmitFilter（Servlet Filter）从 HttpServletRequest Header 读取用户信息，封装成 UserInfoDTO 存入 ThreadLocal（UserContext），Service 层通过 UserContext.getUsername() 获取当前用户。

好处是 Service 层不需要关心如何获取用户信息，统一由 Filter 层注入，代码解耦清晰。

D3. ⭐ 什么是 ThreadLocal？为什么用 ThreadLocal 存用户信息？

答：
ThreadLocal 是 Java 提供的线程本地变量，每个线程拥有独立的变量副本，线程间互不干扰。

用于存用户信息的原因：

  • Web 请求是多线程处理的，每个请求由独立线程处理
  • 将用户信息存入 ThreadLocal 后，同一请求链路中任何位置（Controller → Service → DAO）都可以直接获取，不需要通过方法参数层层传递
  • 请求结束后必须调用 UserContext.removeUser() 清除（本项目在 Filter 的 finally 块中清除），防止内存泄漏和线程池复用时的数据污染

D4. ⭐⭐ 用户限流是怎么做的？

答：
使用 UserFlowRiskControlFilter + Lua 脚本实现滑动窗口限流。以用户名为 key，在指定时间窗口（timeWindow 秒）内统计请求次数，超过 maxAccessCount 则返回限流错误。

使用 Lua 脚本的原因：Redis 执行 Lua 脚本是原子操作，避免了「查询计数 → 判断 → 自增」三步操作之间的竞态条件。配置：时间窗口 1 秒，最大访问次数 20 次。

D5. ⭐⭐ 网关白名单是怎么配置的？哪些接口不需要鉴权？

答：
在网关的 application.yaml 中配置白名单路径列表，TokenValidateGatewayFilterFactory 在过滤时检查请求路径是否在白名单中，在白名单内则直接放行，不检查 Token。

白名单接口包括：

  • /api/short-link/admin/v1/user/login（登录）
  • /api/short-link/admin/v1/user/has-username（检查用户名是否存在）
  • /api/short-link/admin/v1/user（用户注册，POST方法）
  • /api/short-link/v1/（短链接跳转，由 project 服务直接处理）

模块 E：分组管理 & 用户注册

E1. ⭐⭐ gid 是怎么保证全局唯一的？用了哪些手段？

答：
三层保障：

  1. 布隆过滤器（gidRegisterCachePenetrationBloomFilter）：快速判断 gid 是否已存在，O(1) 时间复杂度，不存在则可用
  2. t_group_unique 表唯一索引：尝试向 t_group_unique 插入该 gid，抛出 DuplicateKeyException 说明已存在（布隆过滤器误判），返回 null 重试
  3. Redisson 分布式锁：创建分组时加锁（key = LOCK_GROUP_CREATE:{username}），防止并发创建时同一用户超过最大分组数限制（默认20个）

这是「布隆过滤器 + 数据库唯一索引」双重保障的经典模式。

E2. ⭐ 用户注册时为什么要创建一个默认分组？

答：
短链接必须归属到某个分组（gid 是必填字段），如果用户注册后没有分组就无法创建短链接。因此注册成功后，UserServiceImpl.register() 会自动调用 GroupService.saveGroup(username, "默认分组")，确保每个用户至少有一个可用分组，避免用户注册后第一步就遇到障碍。

E3. ⭐⭐ 短链接分组排序是怎么实现的？

答：
t_group 表中有 sort_order 字段（INT 类型），前端拖拽调整分组顺序时，调用更新接口批量提交新的排序值。查询分组列表时按 ORDER BY sort_order ASC 返回，保证顺序一致。

这种设计的优点是排序逻辑完全在数据库层，不依赖插入顺序，支持任意调整位置。

F1. ⭐⭐ 项目中 admin 调用 project 为什么用 OpenFeign 而不是 RestTemplate？

答：

  • RestTemplate：手动拼接 URL、手动序列化/反序列化，代码冗长
  • OpenFeign：声明式 HTTP 客户端，只需定义接口 + 注解（@FeignClient、@GetMapping 等），框架自动生成实现，与 Spring MVC 风格一致，与 Nacos 服务发现无缝集成，支持负载均衡（lb://service-name）

答：
Nacos 作为服务注册与发现中心：

  • gateway、admin、project 三个服务启动时都向 Nacos 注册自己的 IP 和端口
  • 网关路由配置 uri: lb://short-link-admin 中，lb:// 前缀表示通过 Nacos 服务发现 + 负载均衡找到 admin 服务实例
  • admin 的 OpenFeign 调用 short-link-project 时，也通过 Nacos 获取 project 的真实地址
  • 如果某个服务有多个实例，Nacos + 负载均衡会自动分配请求

F3. ⭐⭐ 项目中 aggregation 模块是做什么的？

答：
aggregation 是聚合部署模块，将 admin 和 project 两个服务打包成一个 JAR 包运行，用于开发调试或资源有限的部署环境。

此时 ShortLinkActualRemoteService 的 url 配置从空（走 Nacos 服务发现）变为 aggregation.remote-url（直连本地），跳过 Nacos，两个服务在同一 JVM 内通过 HTTP 调用（localhost），避免了跨网络开销。

F4. ⭐⭐ 微服务架构的优缺点是什么？

答：

优点：

  • 独立部署：每个服务可以独立发布，不影响其他服务
  • 技术异构：不同服务可以用不同技术栈
  • 故障隔离：一个服务挂掉不影响其他服务（配合熔断降级）
  • 独立扩缩容：高并发的 project 服务可以单独扩容，admin 服务不需要

缺点：

  • 运维复杂：多个服务需要各自部署、监控、日志收集
  • 网络开销：服务间调用走 HTTP/RPC，比进程内调用慢
  • 分布式事务：跨服务的数据一致性难以保证
  • 调试困难：链路追踪需要额外工具（Sleuth/Zipkin）

G4. ⭐ @Transactional 注解用在哪里了？有没有什么注意事项？

答：
项目中 createShortLink()、updateShortLink()、register() 等方法上加了 @Transactional(rollbackFor = Exception.class)，确保短链接主表和路由表的写入要么都成功要么都回滚。

注意事项：

  1. rollbackFor = Exception.class 必须写，否则默认只对 RuntimeException 回滚，受检异常不回滚
  2. 事务方法不能在同类内部调用（会导致 Spring AOP 代理失效，事务不生效）
  3. 事务方法必须是 public，私有方法上的 @Transactional 不生效
  4. 长事务会长时间占用数据库连接，应尽量缩短事务范围，避免在事务内调用外部 API

G8. ⭐⭐ 什么是分布式锁？Redisson 的分布式锁是怎么实现的？

答：
分布式锁是在分布式系统中多个进程/服务之间实现互斥访问的锁机制。

Redisson 分布式锁原理（基于 Lua 脚本 + Redis）：

  1. 加锁：用 SET key value NX PX timeout 原子操作，只有 key 不存在时才设置成功（NX），同时设置超时（PX）防止死锁
  2. 锁续期（看门狗）：Redisson 有 WatchDog 机制，锁持有期间每隔 lockWatchdogTimeout/3 时间自动续期，防止业务未执行完锁就过期
  3. 解锁：用 Lua 脚本原子地检查 value 是否是当前线程持有的，是则删除 key，不是则不操作（防止误解锁他人的锁）
  4. 可重入：Redisson 用 Hash 结构存储锁，field 是线程ID，value 是重入次数，支持同一线程多次加锁

模块 H：项目亮点提炼

H1. ⭐⭐ 你觉得这个项目最大的技术亮点是什么？

答（建议按此思路回答）：

第一：短链接跳转的高并发设计。跳转接口通过布隆过滤器防穿透、空值缓存兜底、Redisson 分布式锁防击穿，形成了完整的缓存保护体系。同时引入 t_link_goto 路由表解决分库分表下跳转无法携带分片键的问题，这个设计比较有深度。

第二：异步统计架构。用 Redis Stream 解耦跳转主流程和统计写入，配合两阶段幂等处理保证统计数据不重不漏，用读写锁解决跨分组修改和统计消费的并发一致性问题，这条链路涉及的技术点比较多。

第三：分库分表的分片键设计。不同的表根据业务查询模式选择不同的分片键（username/gid/fullShortUrl），通过路由表模式解决了因分片键限制导致的跨表查询问题。

H2. ⭐⭐ 这个项目你遇到的最难的问题是什么？怎么解决的？

答（参考回答）：

最难的问题是跨分组修改短链接时的数据一致性问题。

具体情况：t_link 以 gid 分片，修改短链接分组时需要在新旧两个分片上操作数据（旧分片逻辑删除，新分片插入）。与此同时，Redis Stream 消费者可能正在消费这条链接的统计数据，它先查路由表拿 gid，再写统计。如果消费者查到旧 gid 后，修改操作切换了 gid，消费者就会把统计写到错误的分组下。

解决方案：使用 Redisson 读写锁，修改操作持有写锁，统计消费持有读锁，两者互斥，保证消费者拿到的 gid 和最终写入的 gid 一定一致。难点在于识别并发场景下的竞态条件。

H3. ⭐ 如果让你优化这个项目，你会从哪里入手？

答（体现思考深度）：

  1. 统计写入批量化：目前每次跳转发一条消息，消费者逐条处理。高并发时可改为批量消费（每100ms聚合一次），减少数据库写入次数
  2. 布隆过滤器容量规划：随着数据增长，误判率会升高，需要定期监控并考虑扩容或重建策略
  3. Redis 高可用：目前配置的是单节点 Redis，生产环境需要改为 Redis Sentinel 或 Redis Cluster

J1. ⭐⭐ 请讲解一下项目的整体数据库设计思路

答：

项目数据库分为三层设计：用户层、短链接层、统计层。

用户层（admin 服务管理）：

  • t_user：用户表，以 username 分片，存储登录信息（密码 MD5 加密）
  • t_group：分组表，以 username 分片，每个用户有多个分组，sort_order 支持拖拽排序

短链接层（project 服务管理）：

  • t_link：短链接主表，以 gid 分片，存储链接元信息和汇总统计（total_pv/uv/uip）
  • t_link_goto：路由表，以 fullShortUrl 分片，解决跳转时无法携带 gid 分片键的问题

统计层（8张表，均以 fullShortUrl 分片）：

  • t_link_access_stats：聚合统计（按小时、周几）
  • t_link_locale_stats：地区统计
  • t_link_os_stats / t_link_browser_stats / t_link_device_stats / t_link_network_stats：设备环境统计
  • t_link_access_logs：访问明细日志
  • t_link_stats_today：今日统计（用于展示实时数据）

J2. ⭐⭐ 统计表为什么要设计唯一索引？ON DUPLICATE KEY UPDATE 是什么原理？

答：

为什么需要唯一索引：统计数据是累加型的，同一短链接同一时间维度（如2026-03-15 14时）只能有一条记录，多次访问要累加到同一条记录上，而不是插入多条。

ON DUPLICATE KEY UPDATE 原理：

  • 尝试 INSERT，如果唯一索引（full_short_url, date, hour, weekday）冲突
  • 则转为 UPDATE，对 pv/uv/uip 做原子累加
  • 整个操作是原子的，不存在并发问题
  • 比「先 SELECT 再决定 INSERT 或 UPDATE」更高效、更安全

J3. ⭐⭐ t_link 表中为什么同时存 total_pv/uv/uip 和单独的统计表？

答：

两者职责不同：

  • t_link 主表的 total_pv/uv/uip：存储该短链接的历史总计，用于在短链接列表页直接展示总访问量，不需要再去聚合统计表求和，查询性能高
  • 独立统计表（8张）：存储多维度、带时间的明细统计，用于展示统计图表（如按天/小时的趋势图、地区分布饼图、设备占比图）

这是典型的「汇总冗余」设计：主表冗余汇总数据，加速列表查询；明细表保留完整数据，支持多维分析。

J4. ⭐⭐⭐ 短链接的有效期是如何设计的？过期后如何处理？

答：

有效期设计：t_link 表有两个字段：

  • valid_date_type：0=永久有效，1=自定义有效期
  • valid_date：有效期截止时间（valid_date_type=1 时有效）

Redis 缓存 TTL 动态计算（LinkUtil.getLinkCacheValidTime）：

  • 永久有效：TTL = 30天（一个固定的较长时间，避免缓存永不过期占用内存）
  • 自定义有效期：TTL = valid_date - 当前时间（精确到期）
  • 已过期：不写缓存，直接写空值缓存

过期后的处理：

  1. Redis 缓存到期自动失效
  2. 下次访问时，缓存未命中，走完整查询流程
  3. 在查主表步骤（步骤⑨）中检查 valid_date，已过期则写空值缓存并返回 404
  4. 主表中 enable_status 也可以手动禁用链接

J5. ⭐⭐ 访问日志表 t_link_access_logs 有什么用？和统计聚合表有什么区别？

答：

t_link_access_logs（明细日志）：

  • 每次访问插入一条记录，存储 user（Cookie UUID）、ip、os、browser、device、network、locale、create_time
  • 用途：支持精确查询某个用户的访问记录、IP 溯源、异常流量分析
  • 数据量最大（每次跳转一条）

统计聚合表（如 t_link_access_stats）：

  • 每个时间维度只有一条记录，通过 UPSERT 累加
  • 用途：展示趋势图、统计报表，查询性能高
  • 数据量远小于日志表

关系：日志表是原始数据，聚合表是从日志数据衍生出的统计数据（但实际上本项目是消费时同步写入两者，而不是从日志表聚合生成统计表）。

J6. ⭐ 为什么用 del_flag + del_time 两个字段来标记删除，而不是只用 del_flag？

答：

只用 del_flag（0/1）无法配合唯一索引实现「逻辑删除后可重新创建」的需求。

  • 唯一索引 (full_short_url, del_time)：del_time=0 表示未删除，保证有效短链接全局唯一
  • 删除时将 del_time 设置为当前时间戳（而不是改 del_flag），这样同一 fullShortUrl 可以有多条历史删除记录，每条的 del_time 不同，不会冲突
  • del_flag 依然存在用于快速过滤（WHERE del_flag = 0），但唯一约束由 del_time 承担

如果只用 del_flag，则无法同时满足「唯一索引」和「可重复创建删除」两个需求。

模块 K：Redis 进阶 & 数据一致性

K1. ⭐⭐ 如何保证 Redis 和 MySQL 的数据一致性？

答：

这是高频题，核心是选择合适的缓存更新策略。常见方案：

方案一：Cache-Aside（旁路缓存，本项目使用）

  • 读：先读 Redis，命中直接返回；未命中查 MySQL，写入 Redis 后返回
  • 写：先更新 MySQL，再删除 Redis（而不是更新）
  • 为什么删除而不是更新？更新需要额外计算，且在并发写入时容易产生脏数据（两个写请求顺序不确定）；删除是幂等操作，下次读时重建缓存即可

方案二：延迟双删

  • 先删 Redis → 更新 MySQL → 延迟一段时间（如 500ms）再删 Redis
  • 目的：防止在更新 MySQL 期间有请求读到旧值写入 Redis
  • 缺点：延迟时间难以确定，期间仍可能读到旧值

本项目的一致性保证：

  • 短链接更新时先更新 MySQL，再删除对应的 Redis 缓存（goto:{fullShortUrl}）
  • 即使 Redis 删除失败，缓存 TTL 到期后也会自然失效，最终一致
  • 跨分组修改时通过读写锁保证统计数据写入正确的分组，防止脏数据

K2. ⭐⭐ Redis 持久化有哪几种方式？有什么区别？

答：

RDB（Redis Database Backup）：

  • 按照配置的时间间隔对数据做全量快照，生成 .rdb 二进制文件
  • 优点：文件紧凑，恢复速度快，对性能影响小（fork 子进程处理）
  • 缺点：两次快照之间的数据会丢失（最多丢失几分钟数据）

AOF（Append Only File）：

  • 记录每条写命令，追加到 .aof 文件
  • 三种 fsync 策略：always（每条命令）、everysec（每秒，推荐）、no（由 OS 决定）
  • 优点：数据丢失最少（everysec 最多丢 1 秒）
  • 缺点：文件较大，恢复速度慢

混合持久化（Redis 4.0+，推荐）：

  • AOF 文件的前半部分是 RDB 格式的全量数据，后半部分是 AOF 格式的增量数据
  • 兼顾恢复速度和数据完整性

本项目 Redis 存储了布隆过滤器（Bitmap）、用户 Token（Hash）、短链接缓存（String），生产环境建议开启混合持久化。

K3. ⭐⭐ Redis 的过期键删除策略是什么？内存淘汰策略有哪些？

答：

过期键删除策略（两种结合使用）：

  • 惰性删除：访问 key 时才检查是否过期，过期则删除并返回 null。节省 CPU，但可能有大量过期 key 占用内存
  • 定期删除：Redis 每隔一段时间随机抽取一批设置了过期时间的 key，检查并删除已过期的。平衡 CPU 和内存

内存淘汰策略（maxmemory-policy）：

策略 说明
noeviction（默认） 内存满时拒绝写入，返回错误
allkeys-lru 对所有 key 使用 LRU 算法淘汰
volatile-lru 只对设置了过期时间的 key 使用 LRU
allkeys-random 随机淘汰所有 key
volatile-ttl 淘汰剩余 TTL 最小的 key

本项目短链接缓存（goto:{fullShortUrl}）设置了 TTL，建议生产环境配置 volatile-lru，优先淘汰最久未使用的过期 key。

K4. ⭐⭐ Redis 集群有哪几种模式？各有什么适用场景？

答：

主从复制（Master-Slave）：

  • 一主多从，主节点写，从节点读，实现读写分离
  • 缺点：主节点故障需手动切换，无自动故障转移

哨兵模式（Sentinel）：

  • 在主从基础上增加哨兵进程，自动监控主节点，主节点故障时自动选举新主节点
  • 适合：数据量不大但需要高可用的场景
  • 缺点：每个节点存全量数据，无法横向扩展容量

Cluster 模式（Redis Cluster）：

  • 数据分片存储，共 16384 个 slot，分配给各个主节点
  • 每个主节点负责一部分数据，可横向扩展
  • 适合：数据量大、需要高并发写入的场景
  • 缺点：不支持跨 slot 的多 key 操作（如 mget 涉及多个 slot）

本项目目前单节点 Redis，生产环境建议升级为 Sentinel 模式（数据量不大但需要高可用）。

模块 L：MySQL 进阶

L1. ⭐⭐ MySQL 索引的底层数据结构是什么？为什么用 B+ 树而不是 B 树？

答：

MySQL InnoDB 索引底层使用 B+ 树。

B+ 树 vs B 树的区别：

对比点 B 树 B+ 树
数据存储 所有节点（包括内部节点）都存数据 只有叶子节点存数据，内部节点只存索引
叶子节点 无链表连接 叶子节点通过链表连接，支持范围查询
查询稳定性 不稳定（内部节点命中即返回） 稳定（查询路径长度固定为树高）

为什么用 B+ 树：

  1. 内部节点不存数据，单个节点能存更多索引，树更矮，I/O 次数更少
  2. 叶子节点链表支持高效的范围查询（BETWEEN、ORDER BY）
  3. 查询性能稳定，所有查询都需要走到叶子节点

本项目中 t_link 的 (full_short_url, del_time) 唯一索引和 gid 普通索引都是 B+ 树结构。

L2. ⭐⭐ 什么是聚簇索引和非聚簇索引？有什么区别？

答：

聚簇索引（Clustered Index）：

  • InnoDB 中，主键索引就是聚簇索引
  • 叶子节点存储的是完整的行数据
  • 一张表只能有一个聚簇索引（因为数据只能有一种物理排列顺序）
  • 本项目所有表的 id 字段（BIGINT 主键）即聚簇索引

非聚簇索引（Secondary Index）：

  • 也叫二级索引，叶子节点存储的是主键值，而不是完整行数据
  • 通过二级索引查询时，先找到主键，再回表（回到聚簇索引）查完整数据
  • 本项目的 uk_full_short_url_del_time、idx_gid 都是非聚簇索引

回表优化（覆盖索引）：
如果查询的字段都在索引中，不需要回表，称为「覆盖索引」。例如：

如果索引包含 full_short_url、del_time、gid，则直接从索引返回，无需回表。

L3. ⭐⭐ MySQL 事务的四大特性和隔离级别是什么？

答：

ACID 四大特性：

  • A（原子性）：事务内所有操作要么全部成功，要么全部回滚
  • C（一致性）：事务前后数据库从一个合法状态变为另一个合法状态
  • I（隔离性）：并发事务之间互不干扰
  • D（持久性）：事务提交后，数据永久保存（通过 redo log 保证）

四种隔离级别（从低到高）：

隔离级别 脏读 不可重复读 幻读
READ UNCOMMITTED 可能 可能 可能
READ COMMITTED 不会 可能 可能
REPEATABLE READ（MySQL默认） 不会 不会 基本解决
SERIALIZABLE 不会 不会 不会

MySQL InnoDB 默认 REPEATABLE READ，通过 MVCC 解决脏读和不可重复读，通过 Gap Lock（间隙锁）基本解决幻读。

本项目 @Transactional 使用默认隔离级别（REPEATABLE READ）。

L4. ⭐⭐ 什么是 MVCC？InnoDB 是怎么实现它的？

答：

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 InnoDB 实现高并发读写的核心机制，读操作不加锁，通过读历史版本数据来避免读写冲突。

实现原理：

  1. 每行数据有两个隐藏字段：trx_id（最近修改该行的事务ID）、roll_pointer（指向 Undo Log 的指针）
  2. Undo Log：记录数据的历史版本，形成版本链
  3. Read View：事务开始时生成一个快照，记录当前活跃事务列表，用于判断哪个版本对当前事务可见

读取规则：事务读取数据时，沿着版本链找到对当前事务可见的最新版本（trx_id < Read View 中最小活跃事务ID，则可见）。

这就是为什么 REPEATABLE READ 下普通 SELECT 不加锁也不会读到其他事务未提交的数据。

L5. ⭐⭐ 什么是索引失效？哪些情况会导致索引失效？

答：

索引失效是指查询时没有走索引，而是全表扫描，性能急剧下降。

常见导致索引失效的情况：

  1. 对索引列做函数/运算操作：WHERE YEAR(create_time) = 2026 → 失效，应改为范围查询
  2. 使用 != 或 NOT IN：MySQL 优化器认为全表扫描更快时会放弃索引
  3. LIKE 前缀通配符：WHERE short_uri LIKE '%abc' → 失效；LIKE 'abc%' → 有效
  4. 联合索引不遵循最左前缀：索引 (a, b, c)，WHERE b = 1 无法使用该索引
  5. 类型隐式转换：字段是 VARCHAR，查询用数字 WHERE full_short_url = 123，会触发隐式转换导致失效
  6. OR 连接非索引列：WHERE gid = ? OR origin_url = ?，origin_url 无索引则全表扫描

本项目中所有核心查询都精确使用了分片键和索引字段，避免了索引失效问题。

L6. ⭐ 说一下 MySQL 的 redo log 和 binlog 有什么区别？

答：

对比点 redo log binlog
归属 InnoDB 引擎层 MySQL Server 层
作用 保证事务持久性（crash recovery） 主从复制、数据归档
记录内容 物理日志（某页的修改） 逻辑日志（SQL 语句或行变更）
写入时机 事务执行过程中 事务提交时
大小 固定大小，循环写 追加写，可无限增长

两阶段提交：InnoDB 通过「redo log prepare → binlog 写入 → redo log commit」两阶段提交，保证 redo log 和 binlog 的一致性，防止主从数据不一致。

模块 M：Java 基础高频题

M1. ⭐⭐ HashMap 的底层原理是什么？和 ConcurrentHashMap 有什么区别？

答：

HashMap（JDK 8）：

  • 底层：数组 + 链表 + 红黑树
  • 哈希冲突时，同一桶中的元素以链表存储；链表长度超过 8 且数组长度 ≥ 64 时，链表转为红黑树
  • 扩容：负载因子默认 0.75，元素数量超过 capacity × 0.75 时扩容为 2 倍
  • 线程不安全：并发 put 可能导致数据丢失或死循环（JDK 7 的头插法）

ConcurrentHashMap（JDK 8）：

  • 底层结构与 HashMap 相同（数组 + 链表 + 红黑树）
  • 线程安全：对每个桶的头节点使用 synchronized + CAS 操作，锁粒度细化到单个桶
  • 不允许 key 或 value 为 null（HashMap 允许）

本项目没有直接使用 ConcurrentHashMap，但 Redis 操作通过 Redisson 客户端保证了线程安全。

答：

M4. ⭐⭐ 什么是 JVM 内存结构？GC 是怎么工作的？

答：

JVM 运行时内存区域：

  • 堆（Heap）：存放对象实例，是 GC 的主要区域，分为新生代（Eden + S0 + S1）和老年代
  • 方法区/元空间（Metaspace）：存放类信息、常量、静态变量（JDK 8 改为元空间，使用本地内存）
  • 虚拟机栈：每个线程独有，存储方法调用的栈帧（局部变量、操作数栈）
  • 程序计数器：每个线程独有，记录当前执行字节码的行号

GC 流程（分代收集）：

  1. 新对象分配在 Eden 区
  2. Eden 满时触发 Minor GC，存活对象移到 S0/S1，年龄 +1
  3. 年龄超过阈值（默认15）的对象晋升到老年代
  4. 老年代满时触发 Full GC（Stop-The-World，耗时较长）

常见 GC 算法：标记-清除、标记-整理、复制算法（新生代用）；G1（JDK 9+ 默认）、ZGC（低延迟）。

N2. ⭐⭐ TCP 三次握手和四次挥手的过程是什么？为什么是三次而不是两次？

答：

三次握手（建立连接）：

  1. 客户端 → 服务端：SYN（我要连接，seq=x）
  2. 服务端 → 客户端：SYN+ACK（收到了，我也要连接，seq=y, ack=x+1）
  3. 客户端 → 服务端：ACK（收到了，seq=x+1, ack=y+1）

为什么是三次而不是两次：两次握手只能保证服务端知道客户端能发送，但客户端不知道服务端能发送。三次握手确保双方都确认了对方的收发能力。

四次挥手（断开连接）：

  1. 客户端 → 服务端：FIN（我发完了）
  2. 服务端 → 客户端：ACK（收到，但我还有数据要发）
  3. 服务端 → 客户端：FIN（我也发完了）
  4. 客户端 → 服务端：ACK（收到，等待 2MSL 后关闭）

为什么挥手是四次：因为 TCP 是全双工，两个方向的关闭需要独立确认。

O3. ⭐⭐ Spring 如何解决循环依赖？

答：

循环依赖：A 依赖 B，B 依赖 A。

Spring 通过三级缓存解决（仅对 singleton + setter/字段注入有效）：

  1. 一级缓存：存放完全初始化好的 Bean
  2. 二级缓存：存放已实例化但未完成属性注入的 Bean（早期引用）
  3. 三级缓存：存放 Bean 工厂（用于创建代理对象）

解决过程：A 实例化后先放入三级缓存，然后注入 B；B 实例化后注入 A 时，从三级缓存获取 A 的早期引用，完成 B 的初始化；然后 A 继续完成初始化。

不能解决的情况：构造函数注入的循环依赖（因为实例化阶段就需要依赖，放不进缓存）。本项目推荐的 @RequiredArgsConstructor（构造函数注入）如果有循环依赖会直接报错，这反而是好事，强迫开发者重新设计依赖关系。

布隆过滤器 + 空值缓存消除缓存穿透 — 完整讲解

先理解「缓存穿透」是什么问题

正常跳转流程是：查 Redis → 命中则返回，未命中则查数据库。

穿透场景：有人用一个根本不存在的短链接（比如随机生成的 /xxxxxx）发起大量请求，每次 Redis 查不到，每次都去查数据库，数据库也查不到，但数据库承受了所有压力，严重时会被打垮。

布隆过滤器怎么解决

原理：布隆过滤器用多个哈希函数对一个值计算出多个位置，在 Bitmap 中把这些位都标为 1。查询时，如果这些位全是 1，说明可能存在；只要有一位是 0，则一定不存在。

创建短链接时：

bloomFilter.add(“nurl.ink/abc123”) → 在 Bitmap 对应位置标 1

跳转时：

bloomFilter.contains(“nurl.ink/xxxxxx”)

→ 返回 false（某位是0）→ 一定不存在 → 直接返回 404，不查数据库 ✅

→ 返回 true（所有位是1）→ 可能存在 → 继续查 Redis / 数据库

核心特性：

  • 不会漏判：真实存在的短链接，布隆过滤器一定返回 true

  • 有误判率：不存在的短链接，有小概率返回 true（约 3%）

  • 不支持删除：删除短链接后，布隆过滤器里的位不能清除

误判了怎么办？空值缓存兜底

布隆过滤器返回 true，但数据库里查不到（误判 or 链接已过期），这时：

查路由表 → 不存在

→ Redis.set(“goto_is_null:nurl.ink/xxxxxx”, “-“, 30min)

→ 返回 404

下次同样的请求进来：

→ 先查 Redis → 发现空值缓存存在 → 直接返回 404

→ 不再查数据库 ✅

空值缓存 TTL 设为 30 分钟，兜住了布隆过滤器误判 + 已过期链接的场景。

分布式锁 + 双重检查解决缓存击穿 — 完整讲解

先理解「缓存击穿」是什么问题

缓存击穿和穿透不同：

  • 穿透：请求的数据根本不存在（恶意请求）

  • 击穿：数据是真实存在的，但缓存恰好过期了，此时大量并发请求同时涌入，全部去查数据库重建缓存

对于热点短链接（比如某个爆款链接），缓存过期的瞬间可能有几千个请求同时到达，全部打到数据库，数据库扛不住。

为什么不能直接加 synchronized？

synchronized 是 JVM 级别的锁，只在单个服务实例内有效。本项目是微服务，project 服务可以部署多个实例，synchronized 无法跨实例互斥，所以需要分布式锁。

Redisson 分布式锁 + 双重检查的完整流程

关键点：双重检查的意义

锁释放后，后续排队的请求拿到锁时，先不急着查数据库，而是再查一次 Redis。因为前一个请求已经重建了缓存，直接命中返回，不再查数据库。如果没有双重检查，每个拿到锁的请求都会去查数据库，锁就失去了意义。

面试官可能的追问

问：Redisson 分布式锁怎么防止死锁？

> 答：两层保护。第一层是看门狗（WatchDog）机制，锁持有期间每隔 lockWatchdogTimeout/3（默认10秒）自动续期，防止业务还没执行完锁就过期被其他人拿走。第二层是设置锁的最大超时时间兜底，即使看门狗失效，锁也会在超时后自动释放，不会永久死锁。

问：用 lock.lock() 会一直阻塞，会不会影响性能？

> 答：可以用 tryLock(waitTime, leaseTime, unit) 设置等待超时，超过等待时间直接返回失败，给用户返回「系统繁忙」提示，避免线程堆积。本项目学习阶段用的是 lock()，生产环境建议加超时。

问：为什么不用 synchronized 而用分布式锁？

> 答：synchronized 是 JVM 级别的锁，只在单个进程内有效。本项目 project 服务可以横向扩展部署多个实例，多个实例之间 synchronized 无法互斥，只有基于 Redis 的分布式锁才能在多实例间保证同一时刻只有一个请求查数据库。

问：双重检查中第二次查缓存为什么不会出现并发问题？

> 答：Redis 的 GET 操作是原子的，不存在并发读的问题。锁的作用是保证只有一个请求在执行「查库 → 写缓存」这个非原子的复合操作，读缓存本身不需要互斥。

先更新数据库，再删除缓存 — 完整讲解

先理解为什么需要这个策略

短链接支持修改目标地址（originUrl）。比如原来 /abc123 指向百度，现在改指向谷歌。

如果处理不当，用户访问 /abc123 可能还是跳到百度——这就是缓存与数据库不一致的问题。

四种方案对比，说清楚为什么选这个

方案一：先删缓存，再更新数据库

线程A：删除 Redis 缓存

​ 线程B：查缓存 → 未命中 → 查数据库（旧值）→ 写入缓存（旧值）

线程A：更新数据库（新值）

结果：缓存里是旧值，数据库是新值 ❌

存在并发窗口，线程B会把旧值重新写回缓存，导致不一致。

方案二：先更新数据库，再更新缓存

线程A：更新数据库（新值A）

线程B：更新数据库（新值B）

线程B：更新缓存（新值B）

线程A：更新缓存（新值A） ← 后写的反而是旧的

结果：数据库是新值B，缓存是新值A ❌

并发写时顺序无法保证，缓存可能写入错误的值。

方案三：先更新数据库，再删除缓存（本项目使用）✅

线程A：更新数据库（新值）

线程A：删除 Redis 缓存

下次请求：缓存未命中 → 查数据库（新值）→ 写入缓存（新值）✅

删除比更新更安全：删除是幂等操作，不存在「写错值」的问题，下次访问自然重建。

方案四：延迟双删

线程A：删除缓存 → 更新数据库 → 延迟500ms → 再删一次缓存

解决了方案一的并发窗口问题，但延迟时间难以确定，实现复杂，本项目不采用。

本项目的具体实现

下次用户访问该短链接：

查 Redis → 未命中（已删除）

→ 加锁 → 查数据库（新值）→ 重建缓存（新值）

→ 302 跳转到新的目标地址 ✅

这个方案还有没有不一致的窗口？

理论上存在一个极小的并发窗口：

线程A：更新数据库（完成）

​ 线程B：查缓存（未命中）→ 查数据库（读到新值）

线程A：删除缓存（完成）

​ 线程B：写缓存（新值）✅ 这次没问题

但如果 B 查的是数据库提交前的旧值：

线程A：更新数据库（未提交）

线程B：查缓存（未命中）→ 查数据库（读到旧值）→ 写缓存（旧值）

线程A：数据库提交（新值）

线程A：删除缓存 ← 删掉了线程B写的旧值缓存 ✅

只要删缓存在数据库提交之后执行，就能覆盖掉脏缓存。本项目通过 @Transactional 保证数据库先提交，再执行删除缓存，窗口极小且能自愈。

如果删缓存失败了怎么办？

这是这个方案的缺点：数据库更新成功但 Redis 删除失败（网络抖动），缓存会一直是旧值，直到 TTL 过期。

本项目的处理：

  • 短链接缓存有 TTL（根据有效期动态计算），最终会自然过期，保证最终一致性

  • 生产环境可以加重试：删除失败时发一条延迟消息，重试删除

面试官可能的追问

问：这个方案和延迟双删有什么区别，你为什么不用延迟双删？

> 答：延迟双删是「先删缓存 → 更新数据库 → 延迟再删一次」，目的是覆盖掉并发读写产生的脏缓存。我用的「先更新数据库再删缓存」本身就避免了脏缓存写入的问题，不需要第二次删除，实现更简单。延迟双删的延迟时间也难以确定，在延迟期间缓存依然是旧值，不如直接用我的方案。

问：强一致性怎么保证，最终一致性和强一致性有什么区别？

> 答：强一致性要求更新数据库和删除缓存是一个原子操作，任意时刻读到的都是最新值。这在分布式系统中代价极高，需要用到分布式事务（如 2PC）。本项目保证的是最终一致性：允许极短时间内缓存和数据库不一致，但最终（TTL 过期或删除成功后）会恢复一致。对于短链接这类业务，短暂的不一致（几十毫秒到几秒）是可以接受的，不需要强一致性。

问：数据库更新和删除缓存能不能用事务包起来？

> 答：不能。@Transactional 管理的是数据库事务，Redis 操作不在事务范围内。即使数据库回滚，已执行的 Redis 删除操作也无法撤销。正确做法是：事务提交后再删除缓存，本项目就是这样做的——@Transactional 方法内先操作数据库，方法返回前（事务提交后）删除缓存。

面试官可能的追问

问：Redis Stream 和 RocketMQ/Kafka 比有什么劣势？

> 答：三点劣势。第一，可靠性弱于专业 MQ，Redis 本身不是为消息队列设计的，宕机重启可能丢消息（需依赖持久化配置）；第二，不支持死信队列，消费失败的消息处理机制不如 RocketMQ 完善；第三，吞吐量有上限，极高并发场景下 Redis 本身会成为瓶颈。本项目选择 Redis Stream 的原因是不需要额外部署中间件，学习项目够用，生产环境高可靠场景应该用 RocketMQ。

问：消费者挂了消息会丢吗？

> 答：不会。Redis Stream 的消息在被 XDEL 删除之前一直存在。本项目消费者是在成功处理完所有统计写入、标记幂等完成之后才执行 XDEL，消费者重启后未被删除的消息会重新投递，配合幂等机制保证不重复写入。

问：统计数据有延迟，会不会影响用户体验？

> 答：统计数据本身就是异步展示的，用户看统计报表时有秒级延迟是完全可以接受的。跳转功能（核心路径）是即时的，统计功能（非核心路径）异步化，这是合理的取舍。Redis Stream 消费者的处理延迟通常在 1 秒以内，感知不到。

问：为什么构建统计记录时就判断 uvFirstFlag，而不是在消费者里判断？

分库分表路由表设计 — 完整讲解

先理解分库分表的基本原理

ShardingSphere 水平分片的核心逻辑：根据分片键的值，用哈希算法决定数据存在哪张表。

t_link 以 gid 分片，共 16 张表（t_link_0 ~ t_link_15）

分片规则：gid.hashCode() % 16 = 表编号

gid = “default” → hashCode % 16 = 3 → 数据在 t_link_3

gid = “work” → hashCode % 16 = 7 → 数据在 t_link_7

关键约束：ShardingSphere 执行 SQL 时，必须携带分片键才能知道去哪张表查，否则会全路由（16 张表全扫）。

问题：跳转时没有 gid 怎么办

用户访问短链接的完整 URL 是 http://nurl.ink:8001/abc123，浏览器发出的请求只有：

Host: nurl.ink:8001

Cookie: uv=550e8400-…

没有 gid。系统拿到的只有 fullShortUrl = “nurl.ink:8001/abc123”。

如果直接查 t_link：

WHERE full_short_url = ‘nurl.ink:8001/abc123’

AND del_flag = 0

没有 gid 分片键，ShardingSphere 不知道去哪张表，会全路由扫描 16 张表，性能极差，而且每张表数据量千万级，这是不可接受的。

新建一张专门存 fullShortUrl → gid 映射关系的路由表，以 fullShortUrl 为分片键：

– 共 16 张：t_link_goto_0 ~ t_link_goto_15

分片规则：fullShortUrl.hashCode() % 16，跳转时携带 fullShortUrl 查路由表，单分片查询。

面试官可能的追问

问：路由表增加了一次数据库查询，会影响跳转性能吗？

> 答：影响极小，原因有两点。第一，99% 的跳转请求在 Redis 缓存命中阶段就结束了，根本不会走到查路由表这一步；第二，路由表查询是单分片精确查询，走唯一索引，耗时约 1-2ms，可以忽略不计。实际上引入路由表反而比全路由扫描性能高几个数量级。

问：路由表和主表的数据会不会不一致？

> 答：创建时在同一个 @Transactional 事务内写入两张表，要么都成功要么都回滚，不会不一致。跨分组修改时通过 Redisson 写锁保证修改期间统计消费者不会并发读取，修改操作原子地完成「旧分片删除 → 新分片插入 → 路由表更新 gid」三步。

问：为什么不把 gid 存在 Cookie 或者 URL 里，省掉路由表？

> 答：gid 是用户内部的分组标识，不应该暴露给终端用户，URL 里暴露 gid 既有信息泄露风险，也破坏了短链接短小简洁的特性。Cookie 方案同理，而且用户可能清除 Cookie 或换设备，不可靠。路由表是分库分表场景下的经典解法，代价是多一张表，换来的是接口设计的干净和安全。

问：路由表本身数据量很大，会有性能问题吗？

> 答：路由表同样做了分片，以 fullShortUrl 为分片键分了 16 张表，每张表数据量和主表分片一样控制在千万级以内。路由表结构极简（只有 fullShortUrl 和 gid 两个核心字段），查询走唯一索引，不会有性能问题。

> 「鉴权我放在 Gateway 统一做，不在每个业务服务里分别写。流程是：网关收到请求，先检查是否白名单接口，不是的话从 Header 取 username 和 token，去 Redis Hash 里验证，验证通过后把 userId 和 realName 注入到请求 Header 里转发给下游。下游服务有个 Filter 把 Header 里的用户信息存进 ThreadLocal，Service 层直接取用，完全无侵入。限流用 Redis 加 Lua 脚本实现滑动窗口，Lua 脚本保证「查计数、判断、自增、设过期」原子执行，不会有并发漏判的问题。触发限流后不是直接报错，而是返回 429 加友好提示，做了降级处理。」

面试官可能的追问

问：Token 存在 Redis Hash 里，为什么用 Hash 而不是 String？

> 答：Hash 结构 key 是 login:{username}，field 是 token，value 是用户 JSON。这样设计的好处是同一个用户多端登录时，可以存多个 token（不同设备的 field 不同），互不影响；强制下线某个设备时只删对应的 field 即可，不影响其他设备的登录态。如果用 String，每个 token 一个 key，无法方便地管理同一用户的多个登录态。

问：网关是响应式编程（WebFlux），和普通 Servlet 有什么区别？

> 答：Gateway 基于 Spring WebFlux，是非阻塞的响应式编程模型，用 ServerWebExchange 处理请求，修改请求要用 request.mutate().build() 生成新对象（不可变设计）。Admin/Project 服务是普通 Spring MVC（Servlet），用 HttpServletRequest。两者的编程模型不同，这也是为什么网关的 Filter 是 GatewayFilter，而业务服务的 Filter 是 javax.servlet.Filter。

问：ThreadLocal 用完不清除会有什么问题？

> 答：两个问题。第一是内存泄漏：Tomcat 使用线程池，线程请求结束后不销毁而是复用，ThreadLocal 里的数据如果不清除会一直留着，随着请求增多越积越多。第二是数据污染：下一个请求复用该线程时，ThreadLocal 里还有上一个请求的用户信息，导致权限错乱。所以本项目在 UserTransmitFilter 的 finally 块里强制调用 UserContext.removeUser()，无论请求成功还是异常都会清除。

问：如果 Redis 宕机了，鉴权怎么办？

> 答：Redis 宕机时所有 Token 查询失败，用户全部无法访问，这是单点故障。生产环境需要 Redis 高可用方案（Sentinel 或 Cluster）。短期应对可以加熔断降级：Redis 不可用时降级为放行（牺牲安全性保可用性），或者降级为全部拒绝（保安全性牺牲可用性），根据业务重要性选择。

t_link 唯一索引设计 — 完整讲解

先理解业务场景

短链接支持以下操作：

  1. 创建 /abc123 指向百度

  2. 删除 /abc123（移入回收站）

  3. 再次创建 /abc123 指向谷歌（同一个短码重新生成）

这个需求看起来简单，但对数据库唯一约束的设计有挑战。

方案一：只用 full_short_url 做唯一索引（有问题）

UNIQUE KEY uk_full_short_url (full_short_url)

操作流程：

INSERT full_short_url=’nurl.ink/abc123’ → 成功 ✅

​ （记录存在）

DELETE → del_flag=1（逻辑删除，不真删数据）

​ （记录还在，只是 del_flag=1）

INSERT full_short_url=’nurl.ink/abc123’ → ❌ 唯一键冲突！

逻辑删除不移除数据，所以同一 full_short_url 第二次插入直接报错。

方案二：物理删除（有问题）

删除时真正 DELETE 掉这行，第二次创建就没有冲突了。

问题：

  • 统计数据（8 张统计表）依赖 full_short_url 关联，物理删除后统计数据成了孤儿

  • 无法实现回收站功能（删了就找不回来了）

  • 审计日志缺失，无法追溯历史操作

方案三：唯一索引改为 (full_short_url, del_time)（本项目方案）✅

设计思路：把删除时间戳也纳入唯一约束，让「有效记录」和「已删除记录」的唯一键天然不同。

UNIQUE KEY uk_full_short_url_del_time (full_short_url, del_time)

字段约定：

  • del_time = 0：未删除（有效状态）

  • del_time = 当前时间戳：已删除（如 1710000000000）

具体操作流程

第一次创建：

INSERT INTO t_link (full_short_url, del_flag, del_time, …)

VALUES (‘nurl.ink/abc123’, 0, 0, …)

– 唯一键：(‘nurl.ink/abc123’, 0) ✅ 不冲突

删除（移入回收站）：

UPDATE t_link

SET del_flag = 1,

del_time = 1710000000000 – 当前时间戳

WHERE full_short_url = ‘nurl.ink/abc123’

AND del_time = 0 – 只操作有效记录

– 唯一键变为：(‘nurl.ink/abc123’, 1710000000000) ✅

第二次创建：

INSERT INTO t_link (full_short_url, del_flag, del_time, …)

VALUES (‘nurl.ink/abc123’, 0, 0, …)

– 唯一键：(‘nurl.ink/abc123’, 0)

– 历史删除记录的唯一键是 (‘nurl.ink/abc123’, 1710000000000)

– 两者不同，不冲突 ✅

再次删除，再次创建（可无限循环）：

第一条记录：(‘nurl.ink/abc123’, 1710000000000) → 第一次删除

第二条记录：(‘nurl.ink/abc123’, 1710500000000) → 第二次删除

第三条记录：(‘nurl.ink/abc123’, 0) → 当前有效

每条历史删除记录的时间戳都不同，唯一键天然不冲突。

为什么删除时用时间戳而不是用自增ID

方案A：del_time 用当前时间戳（本项目）

优点：业务上有意义，可以知道删除时间

缺点：极小概率同一毫秒删除同一短链接（几乎不可能）

方案B：del_flag 用 0/1，再加一个 del_id 自增

多一个字段，增加复杂度

方案C：del_time 用随机数

没有业务含义，不推荐

用时间戳最优雅：一个字段同时承担「是否删除」和「何时删除」两个语义，且天然保证唯一性。

del_flag 和 del_time 分工

虽然 del_time=0 已经能表示「未删除」，为什么还要保留 del_flag？

– 查有效短链接（高频操作）

WHERE del_flag = 0

– 不能写成：

WHERE del_time = 0 – 虽然等价，但语义不清晰

分工：

  • del_flag：业务语义标识，代码里所有查询用 del_flag = 0 过滤，可读性好，且可以加单列索引加速过滤

  • del_time：承担唯一约束的技术字段，同时记录删除时间，有审计价值

两者互补，缺一不可。

面试口述版（30秒版本）

> 「短链接支持删除后重新创建，如果只用 full_short_url 做唯一索引，逻辑删除后记录还在表里，再次插入就会唯一键冲突。我的设计是把唯一索引改成 full_short_url 和 del_time 的组合索引。del_time=0 表示有效，删除时把 del_time 更新为当前时间戳，这样有效记录的唯一键是 (url, 0)，历史删除记录是 (url, 时间戳)，两者不冲突。第二次创建时 del_time 又是 0，和之前的删除记录时间戳不同，也不冲突。这样就兼容了同一短链接多次创建删除的场景，同时保留了历史数据，回收站功能也能正常使用。」

面试官可能的追问

问：del_time 用时间戳，同一毫秒删除两次同一短链接会不会冲突？

> 答：理论上会，但这个场景在业务上几乎不可能发生——同一个短链接在同一毫秒被删除两次，意味着两个请求在同一毫秒内都通过了「查询有效记录 → 执行删除」的流程，而删除操作本身带了 WHERE del_time = 0 的条件，第一个删除成功后 del_time 已经不是 0 了，第二个删除的 WHERE 条件就匹配不到记录了，所以实际上不会出现两条相同时间戳的删除记录。

问：唯一索引是 (full_short_url, del_time)，那按 full_short_url 查询还走索引吗？

> 答：走。联合索引遵循最左前缀原则，(full_short_url, del_time) 索引可以支持 WHERE full_short_url = ? 的查询（利用索引的左侧列），也支持 WHERE full_short_url = ? AND del_time = ? 的精确查询。跳转时查主表的 SQL 是 WHERE gid = ? AND full_short_url = ? AND del_flag = 0，gid 是分片键决定去哪张表，full_short_url 走联合索引，查询效率很高。

问：这张表数据量会不会越来越大，历史删除记录怎么清理？

> 答：这是个值得思考的问题。短链接删除后对应的统计数据还要保留（用于历史报表），所以不能立即物理删除。可以做定期归档：把 del_time 超过一定时间（如 180 天）的已删除记录迁移到归档表，主表只保留近期数据和有效数据，控制表的数据量。本项目学习阶段没有实现这部分，但这是生产环境必须考虑的问题。

问：为什么不用软删除框架（如 MyBatis-Plus 的逻辑删除）？

> 答：MyBatis-Plus 的逻辑删除通过 @TableLogic 注解把 del_flag=0/1 的过滤自动加到所有查询里，用起来方便。但本项目唯一索引设计不走寻常路，用 del_time 而不是 del_flag 承担唯一约束，而且删除时要同时更新 del_flag 和 del_time 两个字段，MyBatis-Plus 的标准逻辑删除只处理 del_flag，满足不了需求，所以手动控制删除逻辑更灵活。

全路由优化为多单分片查询 — 完整讲解

先理解「全路由扫描」是什么

ShardingSphere 分片的核心规则：SQL 必须携带分片键才能精确路由到对应分片。

t_link 以 gid 分片（16张表）

携带分片键：WHERE gid = ‘default’

→ ShardingSphere 计算：’default’.hashCode() % 16 = 3

→ 只查 t_link_3 ✅（单分片查询）

不携带分片键：WHERE username = ‘zhangsan’

→ ShardingSphere 不知道数据在哪张表

→ 对 t_link_0 ~ t_link_15 全部发送 SQL，结果聚合返回

→ 全路由扫描 ❌（16次数据库查询）

t_link 每张表千万级数据，16 张全扫一遍，性能无法接受。

哪些场景会触发全路由？

场景一：回收站查询

– 查某用户所有已删除的短链接

SELECT * FROM t_link

WHERE del_flag = 1

– 没有 gid，触发全路由 ❌

场景二：统计某用户所有短链接总数

SELECT COUNT(*) FROM t_link

WHERE username = ‘zhangsan’ AND del_flag = 0

– 没有 gid，触发全路由 ❌

场景三：批量操作（如批量禁用某用户所有短链接）

UPDATE t_link SET enable_status = 1

WHERE username = ‘zhangsan’

– 没有 gid，触发全路由 ❌

解决方案：先查 gid 列表，再精确路由

核心思路：

① 先查 t_group（以 username 分片）→ 拿到该用户所有 gid

只需查 1 个分片，因为 t_group 也以 username 分片

② 再用 gid IN (…) 查 t_link

ShardingSphere 对 IN 列表中每个 gid 单独计算分片

→ 每个 gid 对应 1 次单分片查询

→ N 个分组 = N 次精确查询（而非 16 次全扫）

性能对比：

用户有 5 个分组：default、work、study、life、project

全路由方案：16 次查询（每张表全扫）

优化方案：1次查分组 + 5次单分片查询 = 6次查询

假设用户分组分布恰好覆盖 5 个不同分片：

5次精确查询 vs 16次全扫

每次精确查询走主键/索引，每张表只扫少量数据

性能差距：10倍以上

面试口述版（30秒版本）

> 「t_link 以 gid 分片，查某个用户所有短链接时，请求里只有 username 没有 gid，ShardingSphere 不知道数据在哪张表，就会把 SQL 发到全部 16 张表，这叫全路由扫描，性能极差。我的解法是两步走：第一步先查 t_group 表拿到这个用户所有分组的 gid 列表，t_group 以 username 分片，这是单分片查询；第二步用 gid IN (gid1, gid2…) 查 t_link，ShardingSphere 会对 IN 列表里每个 gid 单独计算分片，只查涉及到的那几张表，而不是全部 16 张。一个用户通常有几个到十几个分组，实际查的表数量远小于 16，性能提升显著。回收站、批量操作等场景都用这个方法解决全路由问题。」

面试官可能的追问

问：用户分组很多（比如有 20 个分组），gid IN 里有 20 个值，ShardingSphere 会查多少张表？

> 答：最多查 16 张，因为分片数就是 16。20 个 gid 经过哈希后映射到 0-15 的分片上，必然有多个 gid 落在同一分片，ShardingSphere 会对同一分片的查询合并，实际发出的 SQL 数量 ≤ 16。本项目用户最多创建 20 个分组，极端情况下才会查满 16 张表，正常情况下大多数用户只有几个分组，查的表数量很少。

问：分页查询时，gid IN 多个分片的数据怎么做分页？

> 答：这是分库分表分页的经典难题。ShardingSphere 的做法是：对每个分片执行 LIMIT (pageNumber-1)*pageSize + pageSize，然后在内存中聚合所有结果，重新排序后截取当前页数据。这意味着页数越深（pageNumber 越大），从各分片拉取的数据越多，内存压力越大。本项目短链接管理场景下，用户通常只看前几页，深分页场景很少，可以接受。生产环境深分页问题可以用游标翻页（基于 create_time 的范围查询）代替 OFFSET 分页来优化。

问：如果用户新建了一个分组但还没有短链接，IN 查询会影响性能吗？

> 答：不会。ShardingSphere 路由时只关心 gid 的哈希值决定去哪张表，空分组的 gid 哈希后对应的分片里没有该 gid 的数据，SQL 执行后返回 0 条记录，只是多了一次空查询，对整体性能影响微乎其微。而且空分组的 gid 大概率会和其他有数据的 gid 落在同一分片，SQL 会被合并，不会额外增加查询次数。

问：除了这个方法，还有没有其他解决全路由的思路？

> 答：有两种思路。第一种是冗余分片键：在 t_link 中增加 username 字段，以 (username, gid) 做联合分片，查询时可以带 username 路由。但这会导致分片逻辑复杂，而且 username 和 gid 的分片结果可能冲突，实现难度大。第二种是异构索引：专门建一张以 username 为分片键的短链接索引表，存 username → fullShortUrl 的映射，类似路由表的思路。本项目用先查分组再 IN 查询是最简单直接的方案，利用了已有的 t_group 表，不需要额外的数据结构。
